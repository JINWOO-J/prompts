# Red Team / Blue Team — 공격·방어 관점 분석

## 개념

Red Team(공격자)과 Blue Team(방어자) 관점을 모두 취하여
시스템의 취약점을 찾고 방어 전략을 수립하는 기법.
군사/보안 분야에서 유래. 최근에는 AI 안전성 평가에도 널리 사용된다.

## 핵심 원리

- 적대적 사고: 공격자 관점을 취하면 방어자가 놓치는 취약점 발견
- 균형 잡힌 분석: 공격과 방어 양쪽을 모두 고려하여 현실적 대책 수립
- MITRE ATT&CK 프레임워크와 자연스럽게 결합
- Purple Team(공격+방어 협업)으로 확장 가능


## 실전 예시

**시나리오: "쿠버네티스 클러스터 보안 평가"**

Red Team:
1. 공개된 API 서버 → 인증 없이 kubectl 접근 시도 (난이도: 하, 영향: 상)
2. 컨테이너 탈출 → privileged 컨테이너에서 호스트 접근 (난이도: 중, 영향: 상)
3. RBAC 우회 → 과도한 권한의 ServiceAccount 악용 (난이도: 중, 영향: 중)

Blue Team:
1. API 서버 → Network Policy + OIDC 인증 적용 (P1)
2. 컨테이너 → PodSecurityPolicy/Standards 적용, privileged 금지 (P1)
3. RBAC → 최소 권한 원칙 적용, 정기 감사 (P2)

## 변형 및 조합

- **Red/Blue + STRIDE**: 위협 모델링 프레임워크와 결합
- **Red/Blue + OODA**: 공격/방어 각각에 OODA 루프 적용
- **Red/Blue + Pre-mortem**: 공격 성공 시나리오를 Pre-mortem으로 분석

## 주의사항

- 실제 공격 시뮬레이션 없이 "상상"만으로는 한계가 있음
- 공격자의 동기와 능력을 현실적으로 가정해야 함
- 방어 비용 대비 효과를 고려해야 함 (모든 공격을 막을 수는 없음)
- 내부자 위협은 별도로 고려해야 함

## 적합한 상황

보안 감사, 시스템 리뷰, 장애 시나리오 분석, 위협 모델링, AI 안전성 평가
