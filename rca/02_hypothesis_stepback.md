---
기법: Step-back 프롬프팅 + 가설 검증 (Hypothesis Testing)
난이도: 중급
적합 상황: 원인이 불명확한 장애, 데이터 부족, 초기 원인 범주 좁히기
필수 입력: 장애 현상 요약, 발생 시각, 시스템 구성, 관찰된 증상 목록
예상 출력: 상위 원인 패턴 분석, 2~3개 가설 + 각 가설 검증에 필요한 데이터 목록
---

# 프롬프트 02: 가설 수립 + Step-back 분석

## 사용 방법
`[대괄호]` 안의 내용을 실제 인시던트 정보로 교체하여 사용.  
**01_basic_rca.md로 분석했는데 원인이 불명확할 때**, 또는 **초기 단계에서 방향을 잡을 때** 활용.

---

## 프롬프트 본문 (Step 1: Step-back)

```
<Role>
당신은 10년 경력의 시니어 SRE 및 분산 시스템 전문가입니다.
세부 로그 분석 전에 문제의 상위 맥락과 일반적인 패턴을 먼저 파악하는 Step-back 방식으로 접근합니다.
</Role>

<Context>
- 서비스/시스템: [서비스명 또는 시스템 이름]
- 기술 스택: [예: MSA on AWS, Redis + RDS + ECS]
- 장애 발생 시각: [YYYY-MM-DD HH:MM KST]
- 관찰된 증상: [예: 특정 API 타임아웃, DB 커넥션 풀 고갈 경고, 메모리 사용량 급증]
</Context>

<Task>
**[Step 1 - Step-back 질문]**
구체적인 로그 분석에 앞서, 다음 질문에 먼저 답하세요:

"[기술 스택 또는 시스템 유형]에서 [관찰된 증상]이 나타날 때, 일반적으로 발생하는 원인 패턴의 카테고리는 무엇인가?
각 카테고리별로 흔히 관찰되는 징후(Leading Indicator)를 함께 설명하라."

이 Step-back 분석 결과를 토대로 다음 Step 2를 진행하세요.
</Task>

<OutputFormat>
## Step 1: 상위 원인 패턴 분석
| 원인 카테고리 | 설명 | 흔한 징후 |
|-------------|------|---------|
| | | |
| | | |
| | | |
</OutputFormat>
```

## 프롬프트 본문 (Step 2: 가설 수립 및 검증 계획)

> Step 1 출력 결과를 아래 `<StepbackResult>`에 붙여넣고 실행하세요.

```
<Role>
당신은 10년 경력의 시니어 SRE입니다. Step-back 분석 결과를 바탕으로
이 인시던트의 구체적인 가설을 수립하고 각 가설의 검증 계획을 작성합니다.
반드시 첨부된 정보 내에서 분석하고, 불확실한 내용은 "데이터 부족"으로 표기하세요.
</Role>

<StepbackResult>
[Step 1 출력 결과를 여기에 붙여넣기]
</StepbackResult>

<ObservedData>
**현재 확보된 데이터:**
[확보된 로그, 메트릭, 알림 내용 등을 여기에 붙여넣기. 없으면 "없음"으로 표기]
</ObservedData>

<Task>
Step-back 분석과 현재 확보된 데이터를 기반으로:

1. 2~3개의 구체적인 원인 가설을 수립하세요.
2. 각 가설에 대해:
   - 가설이 맞다면 어떤 데이터/지표/로그에서 어떤 패턴이 나타나야 하는가?
   - 가설이 틀렸다면 어떤 데이터/지표/로그에서 어떤 패턴이 나타나야 하는가?
3. 확인 우선순위를 지정하세요.
</Task>

<OutputFormat>

  - 확인 대상: [로그/메트릭/지표]
  - 예상 패턴: [어떤 값 또는 패턴이 보여야 하는가]
```
