# KubeVersionMismatch

## 의미

클러스터 전체에서 서로 다른 시맨틱 버전의 Kubernetes 컴포넌트가 실행되고 있으며(KubeVersionMismatch 알림 트리거), Control Plane 노드, Worker 노드 또는 컴포넌트가 서로 다른 Kubernetes 버전을 실행하고 있습니다. 일반적으로 클러스터 업그레이드 과정에서 발생합니다. 클러스터 대시보드에서 노드가 서로 다른 Kubernetes 버전을 보이고, Control Plane 컴포넌트에서 버전 불일치가 나타나며, 로그에 API 버전 호환성 오류가 나타날 수 있습니다. 이는 Control Plane과 Data Plane에 영향을 미치며, API 버전 불일치, 기능 가용성 문제 또는 컴포넌트 통신 문제를 유발할 수 있는 버전 비호환성을 나타냅니다. 일반적으로 불완전한 업그레이드, 실패한 업그레이드 프로세스 또는 수동 버전 변경이 원인이며, 애플리케이션에서 API 버전 호환성 오류가 발생할 수 있습니다.

## 영향

KubeVersionMismatch 알림이 발생합니다. Kubernetes 컴포넌트 간의 호환되지 않는 API 버전이 문제를 유발할 수 있습니다. 단일 컨테이너, 애플리케이션 또는 클러스터 안정성에 영향을 미칠 수 있습니다. API 버전 불일치가 발생하고, 컴포넌트 간 기능 가용성이 다릅니다. 컴포넌트 통신이 실패할 수 있고, 클러스터 운영이 일관되지 않을 수 있습니다. 진행 중인 Kubernetes 업그레이드 프로세스가 불완전하거나 실패했을 수 있으며, 버전별 API 폐기 또는 변경이 호환성 문제를 유발합니다.

## 플레이북

1. 모든 노드를 wide 출력으로 나열하여 Kubernetes 버전을 포함한 모든 노드를 조회하고 클러스터 전체의 버전 불일치를 식별합니다.

2. kube-system 네임스페이스에서 마지막 타임스탬프 순으로 이벤트를 나열하여 최근 Control Plane 이벤트를 조회하고, 버전 관련 오류, API 호환성 문제 또는 업그레이드 실패를 필터링합니다.

3. kube-system 네임스페이스에서 tier=control-plane 레이블을 가진 Pod를 describe하여 API Server, Controller Manager, Scheduler를 포함한 상세 Control Plane 컴포넌트 정보를 확인하고 컴포넌트 버전을 점검합니다.

4. 클러스터 업그레이드 상태를 조회하고 진행 중인 Kubernetes 업그레이드 프로세스가 있는지 확인합니다. 특히 관리형 서비스에서 불일치가 업그레이드 중 예상되는 것인지 판단합니다.

5. 컴포넌트 버전을 Kubernetes 호환성 매트릭스와 비교하여 호환성에 영향을 미칠 수 있는 버전별 API 폐기 또는 변경을 확인합니다.

6. 서로 다른 버전을 실행하는 `<namespace>` 네임스페이스의 Pod `<pod-name>`에서 이벤트와 로그를 조회하고 버전 관련 오류를 필터링하여 호환성 문제를 식별합니다.

## 진단

1. 플레이북의 클러스터 및 노드 이벤트를 분석하여 업그레이드가 진행 중이거나 최근에 발생했는지 식별합니다. 이벤트에서 업그레이드 관련 활동이 나타나면, 업그레이드 기간 동안 버전 불일치가 예상될 수 있습니다.

2. 이벤트에서 활성 업그레이드 프로세스가 나타나면, 플레이북 4단계의 업그레이드 상태를 검증합니다. 이벤트 타임스탬프에서 업그레이드가 여전히 진행 중이면, 버전 불일치는 예상되며 완료 시 해결됩니다.

3. 이벤트에서 업그레이드 실패 또는 롤백 활동이 나타나면, 업그레이드 로그에서 오류를 검토합니다. 이벤트에서 실패한 업그레이드 시도 또는 불완전한 노드 업그레이드가 나타나면, 업그레이드 프로세스에 수정이 필요합니다.

4. 이벤트에서 노드 업그레이드 또는 교체 활동이 나타나면, 플레이북 1단계의 이벤트 타임스탬프와 노드 버전 변경을 연관시킵니다. 업그레이드 이벤트 후 특정 노드가 다른 버전을 보이면, 해당 노드가 업그레이드에 실패했을 수 있습니다.

5. 이벤트에서 API 호환성 오류가 나타나면, 어떤 컴포넌트 버전이 호환되지 않는지 식별합니다. 특정 타임스탬프에서 이벤트가 컴포넌트 간 API 버전 오류를 보이면, 버전 스큐가 즉각적인 주의가 필요한 운영 문제를 유발하고 있습니다.

6. 이벤트에서 노드 전체의 체계적인 버전 패턴(Control Plane vs Worker)이 나타나면, 버전 스큐가 예상된 업그레이드 순서를 따르고 있습니다. 버전 불일치가 특정 노드에 국한되면, 해당 노드에 대한 개별 조사가 필요합니다.

7. 이벤트에서 업그레이드 활동이 나타나지 않으면, 버전 불일치가 수동 개입 또는 구성 드리프트로 인한 것인지 확인합니다. 업그레이드 이벤트 없이 버전이 다르면, 수동 변경 또는 잘못된 구성이 원인입니다.

**연관 관계를 찾을 수 없는 경우**: 업그레이드 프로세스에 대해 시간 범위를 24시간으로 확장하고, 클러스터 업그레이드 절차를 검토하며, 실패한 업그레이드 작업을 확인하고, 노드 풀 버전 구성을 검증하며, 과거 버전 일관성 패턴을 검토합니다. 버전 불일치는 즉각적인 운영 변경보다 불완전한 업그레이드, 수동 버전 변경 또는 업그레이드 프로세스 실패로 인해 발생할 수 있습니다.
