# NACL Blocking Expected Traffic - NACL 예상 트래픽 차단

## 의미

Network ACL(NACL) 규칙이 예상 트래픽을 차단하는 현상(연결 실패 또는 VPCTrafficBlocked 경보 트리거)은 NACL 인바운드 또는 아웃바운드 규칙이 필요한 트래픽을 거부하거나, 규칙 순서가 트래픽 흐름을 방해하거나, 임시 포트 범위가 허용되지 않거나, NACL 규칙 변경이 네트워크 접근을 제한하거나, 서브넷 연결 변경이 트래픽 흐름에 영향을 미칠 때 발생합니다.
 네트워크 트래픽이 차단되고, 리소스 간 연결이 실패하며, 애플리케이션이 통신할 수 없습니다. 이는 네트워킹 계층에 영향을 미치며 서비스 통신을 차단합니다. 일반적으로 NACL 규칙 잘못된 구성, 규칙 순서 문제 또는 임시 포트 규칙 누락이 원인이며, 보안 그룹과 함께 NACL을 사용하는 경우 규칙 평가 순서가 트래픽에 영향을 미칠 수 있고 애플리케이션에서 연결 실패가 발생할 수 있습니다.

## 영향

네트워크 트래픽이 차단됩니다. 리소스 간 연결이 실패합니다. 애플리케이션이 통신할 수 없습니다. NACL 규칙이 예상 트래픽을 거부합니다. 네트워크 접근이 제한됩니다. 서비스 간 통신이 실패합니다. 사용자 대면 연결 오류가 발생합니다. 네트워크 보안 잘못된 구성이 기능에 영향을 미칩니다. VPCTrafficBlocked 경보가 발생할 수 있으며, 보안 그룹과 함께 NACL을 사용하는 경우 규칙 평가 순서가 트래픽에 영향을 미칠 수 있습니다. 차단된 연결로 인해 애플리케이션에서 오류나 성능 저하가 발생할 수 있으며, 서비스 간 통신이 완전히 차단될 수 있습니다.

## 플레이북

1. Network ACL `<nacl-id>`가 존재하고 리전 `<region>`의 VPC AWS 서비스 상태가 정상인지 확인합니다.
2. 리전 `<region>`의 Network ACL `<nacl-id>`를 조회하여 인바운드 및 아웃바운드 규칙, 규칙 순서, 규칙 작업(허용/거부)을 점검하고, 규칙 평가 순서를 검증합니다.
3. VPC Flow Logs가 포함된 CloudWatch Logs 로그 그룹에서 Network ACL `<nacl-id>` 관련 거부된 트래픽 패턴을 필터링하여 거부된 트래픽의 소스와 대상을 확인합니다.
4. Network ACL `<nacl-id>`와 연관된 VPC 서브넷의 CloudWatch 지표(패킷 드롭 지표)를 최근 1시간 동안 조회하여 차단된 트래픽 패턴을 파악하고, 드롭 패턴을 분석합니다.
5. Network ACL `<nacl-id>`와 연관된 VPC 서브넷을 나열하고 NACL 규칙과 상호작용할 수 있는 서브넷 라우팅 및 보안 그룹 구성을 확인하여 서브넷 연결을 검증합니다.
6. CloudTrail 이벤트가 포함된 CloudWatch Logs 로그 그룹에서 Network ACL `<nacl-id>` 관련 NACL 규칙 변경 이벤트를 필터링하여 규칙 변경 사항을 확인합니다.
7. Network ACL `<nacl-id>`의 서브넷 연결을 조회하여 어떤 서브넷이 이 NACL을 사용하는지 확인하고, 서브넷 연결 변경이 트래픽에 영향을 미쳤는지 점검합니다.
8. VPC 서브넷의 CloudWatch 지표(패킷 수)를 조회하여 트래픽 흐름 패턴을 확인하고, 트래픽이 NACL 수준에서 차단되고 있는지 점검합니다.
9. VPC Flow Logs가 포함된 CloudWatch Logs 로그 그룹에서 허용된 트래픽 패턴을 필터링하여 트래픽이 예상 패턴과 일치하는지 확인하고, 규칙 효과를 점검합니다.

## 진단

1. VPC Flow Logs(플레이북 3단계)를 분석하여 트래픽 거부가 처음 나타난 시점을 파악합니다. REJECT 작업의 타임스탬프가 차단이 시작된 시점과 영향을 받는 트래픽 패턴을 확립합니다.

2. CloudTrail에서 거부 타임스탬프 전후로 NACL 규칙 변경(플레이북 6단계)이 표시되면 해당 규칙 변경이 근본 원인일 가능성이 높습니다. 변경 전후 규칙 구성을 비교합니다.

3. NACL 규칙(플레이북 2단계)에서 동일한 트래픽에 대해 Allow 규칙보다 낮은 규칙 번호의 Deny 규칙이 있으면 규칙 순서가 트래픽을 방해하고 있는 것입니다. NACL은 오름차순으로 규칙을 평가하고 첫 번째 일치에서 중지합니다.

4. 아웃바운드 트래픽이 차단되지만 인바운드 규칙이 올바르면 임시 포트 구성을 확인합니다. 반환 트래픽은 임시 포트(1024-65535)를 사용하며 NACL에서 명시적 아웃바운드 Allow 규칙이 필요합니다.

5. 거부 패턴(플레이북 3단계)이 특정 서브넷에 영향을 미치면 서브넷 연결(플레이북 7단계)을 확인합니다. 최근 서브넷 연결 변경으로 예상과 다른 NACL이 적용되었을 수 있습니다.

6. Flow Logs(플레이북 9단계)에서 유사한 패턴의 일부 흐름은 허용되지만 다른 흐름은 거부되면 특정 소스/대상 조합을 확인합니다. NACL이 예상치 못한 트래픽을 올바르게 차단하면서 합법적인 트래픽도 차단하고 있을 수 있습니다.

7. 패킷 드롭 지표(플레이북 4단계 및 8단계)가 거부 패턴과 상관관계를 보이면 보안 그룹이나 라우트 테이블이 아닌 NACL이 차단 계층임을 확인합니다.

상관관계를 찾을 수 없는 경우: 분석 기간을 4시간으로 확장하고, 추가 제한에 대한 보안 그룹 규칙을 검토하고, 라우팅 문제에 대한 라우트 테이블 구성을 확인하고, NAT 게이트웨이 또는 인터넷 게이트웨이 연결을 검증하고, NACL 규칙 할당량 제한을 점검합니다.
