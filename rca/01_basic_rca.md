# 프롬프트 01: 기본 RCA (5-Whys + CoT)

## 사용 방법
`[대괄호]` 안의 내용을 실제 인시던트 정보로 교체한 뒤 LLM에 붙여넣기.

---

## 프롬프트 본문

```
<Role>
당신은 10년 경력의 시니어 SRE(Site Reliability Engineer)입니다.
인프라 장애 분석 및 근본 원인 도출 전문가로서 체계적인 RCA를 수행합니다.
반드시 첨부된 정보 내에서만 분석하고, 불확실한 내용은 "데이터 부족 - 추가 확인 필요"로 명시하세요.
</Role>

<Context>
- 서비스/시스템: [서비스명 또는 시스템 이름]
- 기술 스택: [예: Kubernetes on GCP, Node.js API, PostgreSQL]
- 장애 발생 시각: [YYYY-MM-DD HH:MM KST]
- 장애 종료/복구 시각: [YYYY-MM-DD HH:MM KST] (미복구 시 "진행 중" 기재)
- 영향 범위: [영향 받은 사용자 수, 기능, 리전 등]
</Context>

<Task>
아래 장애 현상과 로그를 기반으로 5-Whys 기법과 Chain-of-Thought 방식을 사용하여 단계별 근본 원인 분석을 수행하세요.

**장애 현상:**
[장애 현상을 구체적으로 기술. 예: "결제 API 응답 시간이 평균 200ms에서 8000ms로 증가하며 5xx 에러율이 35%에 도달"]

**관련 로그/에러 메시지:**
[로그를 여기에 붙여넣기. 없으면 "로그 미제공"으로 명시]
</Task>

<OutputFormat>
다음 형식으로 출력하세요:

## 1. 장애 요약
- 현상:
- 영향:
- 발생 시각:

## 2. 5-Whys 분석
**Why 1:** [관찰된 현상은 왜 발생했는가?]
→ 답변:

**Why 2:** [Why 1의 답변은 왜 발생했는가?]
→ 답변:

**Why 3:** [Why 2의 답변은 왜 발생했는가?]
→ 답변:

**Why 4:** [Why 3의 답변은 왜 발생했는가?]
→ 답변:

**Why 5:** [Why 4의 답변은 왜 발생했는가?]
→ 답변:

## 3. 근본 원인 (Root Cause)
[최종 도출된 근본 원인을 1~2문장으로 명시]

## 4. 기여 요인 (Contributing Factors)
- [근본 원인 외에 장애를 악화시킨 요소들]

## 5. 재발 방지 권고안
| 우선순위 | 액션 | 담당 | 기한 |
|---------|------|------|------|
| 높음 | | | |
| 중간 | | | |
| 낮음 | | | |

## 6. 불확실 항목 (추가 확인 필요)
- [데이터 부족으로 분석하지 못한 항목]
</OutputFormat>
```

---

## 활용 팁

- **Why 5회가 부족할 경우**: "Why 6", "Why 7"로 계속 진행해도 됩니다. 프로세스/정책 실패 수준까지 도달하는 것이 목표입니다.
- **로그가 없는 경우**: `<Task>` 하단에 "현재 확보된 로그가 없으므로 가능한 원인 가설을 제시하고, 각 가설 검증을 위해 확인해야 할 로그/지표를 나열하라"고 추가하세요.
- **다음 단계**: 원인이 불명확하면 `02_hypothesis_stepback.md`로 이행하세요.
