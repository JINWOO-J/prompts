# Self-Refinement — 반복 자기 개선

## 개념

LLM이 자신의 출력을 비평(critique)하고, 그 피드백을 바탕으로
반복적으로 개선하는 기법. Madaan et al. (2023) "Self-Refine" 논문에서 체계화.
Generate → Critique → Refine 사이클을 수렴할 때까지 반복한다.

## 핵심 원리

- 반복적 개선: 한 번에 완벽한 답을 내기 어려우므로 점진적으로 품질 향상
- 자기 비평 능력: GPT-4급 모델은 자신의 출력을 비평하는 능력이 상당히 높음
- 수렴성: 보통 2-3회 반복이면 품질이 수렴 (그 이상은 개선 미미)
- 코드 생성, 수학, 추론 등 다양한 태스크에서 5-20% 성능 향상

## 프롬프트 템플릿

```
**초기 출력 (Generate):**
[질문/작업]에 대한 첫 번째 답변:
[초기 답변]

**자기 비평 (Critique):**
위 답변을 다음 기준으로 평가하세요:
1. 정확성: [정확한가? 오류는?]
2. 완전성: [빠진 내용은?]
3. 명확성: [이해하기 쉬운가?]
4. 실용성: [실제로 적용 가능한가?]

구체적 문제점:
- [문제점 1]: [왜 문제인지, 어떻게 개선할지]
- [문제점 2]: [왜 문제인지, 어떻게 개선할지]

**개선된 출력 (Refine):**
[비평을 반영하여 수정된 답변]

**2차 비평 (필요 시):**
[추가 문제점이 있는가?]
- 있다면: [문제점과 개선 방향]
- 없다면: "품질 충분 — 최종 확정"

**최종 출력:**
[최종 답변]
```

## 실전 예시

**시나리오: Dockerfile 최적화**

초기: 멀티스테이지 빌드 없이 모든 빌드 도구가 최종 이미지에 포함 (800MB)

비평:
1. 이미지 크기 과대 — 빌드 도구 불필요
2. 레이어 캐시 비효율 — COPY . . 이 너무 일찍 실행
3. 보안 — root 사용자로 실행

개선: 멀티스테이지 빌드 + 의존성 먼저 COPY + non-root USER → 120MB

2차 비평: .dockerignore 누락 → 추가 → 최종 확정

## 변형 및 조합

- **Self-Refinement + CoVe**: 비평 시 사실 검증도 포함
- **Self-Refinement + Role**: "시니어 리뷰어로서 비평하세요"
- **Self-Refinement + Rubric**: 평가 기준표를 명시하여 체계적 비평
- **Self-Refinement + Few-Shot**: 좋은 비평/개선 예시 제공

## 주의사항

- 약한 모델은 자기 비평 능력이 부족하여 오히려 품질이 하락할 수 있음
- 3회 이상 반복하면 "과적합" — 불필요한 수정이 추가될 수 있음
- 비평 기준이 모호하면 주관적/일관성 없는 피드백이 나옴
- 원래 답변이 이미 좋은 경우 불필요한 오버헤드

## 적합한 상황

글쓰기, 코드 작성, 프롬프트 개선, 문서 작성, 품질이 중요한 모든 작업
