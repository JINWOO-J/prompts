# OODA Loop — 관찰·판단·결정·행동 루프

## 개념

Observe(관찰) → Orient(판단) → Decide(결정) → Act(행동)의 반복 루프.
존 보이드(John Boyd) 대령이 공중전 의사결정을 위해 개발.
상대보다 빠르게 OODA 루프를 돌리는 것이 승리의 핵심.
인시던트 대응, 애자일 운영, 경쟁 전략에 널리 적용된다.

## 핵심 원리

- 속도 우위: 상대(또는 문제)보다 빠르게 루프를 돌리면 주도권 확보
- Orient(판단)이 핵심: 단순 관찰이 아닌 맥락 해석이 의사결정 품질을 좌우
- 암묵적 가이드: 반복을 통해 Orient → Act가 자동화되면 Decide 단계 생략 가능 (직관)
- 피드백 루프: Act의 결과가 다음 Observe로 이어져 지속적 적응

## 프롬프트 템플릿

```
**상황:** [현재 상황 설명]

**🔍 Observe (관찰) — "무엇이 보이는가?"**
- 현재 데이터/지표: [메트릭, 로그, 알림 등]
- 변화 감지: [이전 대비 무엇이 달라졌는가]
- 외부 신호: [사용자 보고, 의존 서비스 상태 등]
- 정보 공백: [아직 모르는 것]

**🧭 Orient (판단) — "이것이 무엇을 의미하는가?"**
- 데이터 해석: [관찰 결과의 의미]
- 과거 경험과 비교: [유사한 상황이 있었는가]
- 멘탈 모델 적용: [어떤 패턴에 해당하는가]
- 문화적/조직적 맥락: [팀/조직의 제약 고려]
- 가설: [가장 유력한 설명]

**🎯 Decide (결정) — "무엇을 할 것인가?"**
- 선택지: [A / B / C]
- 각 선택지의 예상 결과: [...]
- 선택: [X]
- 근거: [Orient 단계의 판단에 기반]
- 되돌릴 수 있는가: [가역적/비가역적]

**⚡ Act (행동) — "실행하고 관찰하라"**
- 즉시 실행: [구체적 행동]
- 실행 결과 모니터링: [어떤 지표를 볼 것인가]
- 다음 OODA 사이클 트리거: [어떤 조건에서 다시 관찰을 시작하는가]
- 타임박스: [다음 판단까지 최대 대기 시간]
```

## 실전 예시

**시나리오: "프로덕션 레이턴시 급증 인시던트"**

Observe: P99 레이턴시 200ms → 5s, 에러율 0.1% → 15%, 최근 배포 10분 전
Orient: 배포 직후 발생 → 새 코드가 원인일 가능성 높음. 과거에도 유사 패턴 있었음
Decide: 즉시 롤백 (가역적, 리스크 낮음) vs 핫픽스 (시간 소요, 리스크 높음) → 롤백 선택
Act: 롤백 실행 → 레이턴시 정상화 확인 → 다음 OODA: 원인 분석 후 수정 배포

→ 전체 루프 소요 시간: 5분 (빠른 OODA가 MTTR 단축)

## 변형 및 조합

- **OODA + ReAct**: Observe-Orient를 Thought로, Act를 Action으로 매핑
- **OODA + 5 Whys**: Orient 단계에서 5 Whys로 근본 원인 파악
- **OODA + Hypothesis**: Orient에서 가설 수립, Act에서 검증

## 주의사항

- Orient(판단) 단계를 건너뛰면 잘못된 행동으로 이어짐
- 정보가 불완전한 상태에서 결정해야 하는 경우가 많음 (70% 정보로 결정)
- 루프가 너무 빠르면 충분한 분석 없이 행동할 수 있음
- 팀 OODA는 개인 OODA보다 느림 — 커뮤니케이션 오버헤드

## 적합한 상황

인시던트 대응, 실시간 의사결정, 경쟁 전략, 애자일 운영, 온콜 대응, 위기 관리
