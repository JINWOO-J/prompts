# ACM 인증서 발급 실패 — AWS Certificate Manager (ACM) Certificate Not Issuing

## 의미

ACM 인증서가 발급되지 않습니다(인증서 프로비저닝 실패 또는 ACMCertificateIssuanceFailed 알람 발생). 원인으로는 도메인 검증 실패, DNS 검증 레코드 오류, 이메일 검증 미완료, 인증서 요청 오류, 발급 중 인증 기관 문제, ACM 인증서 요청 구성 오류 등이 있습니다. ACM 인증서가 발급되지 않고, SSL/TLS 인증서를 프로비저닝할 수 없으며, 인증서 검증이 실패합니다. 이는 보안 및 인증서 관리 계층에 영향을 미치며 보안 연결을 차단합니다. 일반적으로 검증 문제, DNS 구성 문제, 요청 오류가 원인이며, CloudFront 또는 API Gateway와 함께 ACM을 사용하는 경우 인증서 검증 요구사항이 다를 수 있고 애플리케이션에서 인증서 프로비저닝 실패가 발생할 수 있습니다.

## 영향

ACM 인증서 미발급, SSL/TLS 인증서 프로비저닝 불가, 인증서 검증 실패, 도메인 검증 미완료, 인증서 요청 보류 상태 유지, 인증서 자동화 실패, 인증서 발급 알람 발생, 보안 연결 수립 불가. ACMCertificateIssuanceFailed 알람 발생 가능. CloudFront 또는 API Gateway와 함께 ACM을 사용하는 경우 인증서 검증 요구사항이 다를 수 있음. 인증서 부재로 인해 애플리케이션 오류나 성능 저하 발생 가능. 보안 HTTPS 연결을 수립할 수 없습니다.

## 플레이북

1. ACM 인증서 `<certificate-arn>`의 존재를 확인하고 리전 `<region>`의 ACM AWS 서비스 상태가 정상인지 확인합니다.
2. 리전 `<region>`의 ACM 인증서 `<certificate-arn>`을 조회하여 인증서 상태, 도메인 검증 상태, 검증 방법, 인증서 요청 세부사항을 검사하고 검증 상태를 확인합니다.
3. ACM 이벤트가 포함된 CloudWatch Logs 로그 그룹을 조회하여 인증서 `<certificate-arn>`과 관련된 인증서 검증 실패 이벤트나 발급 오류 패턴을 필터링하고 검증 오류 세부사항을 포함합니다.
4. 검증 시도 패턴을 포함한 ACM 인증서 검증에 대한 CloudWatch 메트릭을 지난 7일 동안 조회하여 검증 문제를 식별하고 검증 시도를 분석합니다.
5. 인증서 `<certificate-arn>`의 ACM 인증서 도메인 검증 레코드를 나열하고 DNS 레코드 구성 또는 이메일 검증 상태를 확인하며 검증 레코드를 확인합니다.
6. DNS 검증에 Route 53을 사용하는 경우 Route 53 호스팅 영역 `<hosted-zone-id>`를 조회하여 DNS 검증 레코드가 올바르게 구성되었는지 확인하고 레코드 존재를 확인합니다.
7. CloudTrail 이벤트가 포함된 CloudWatch Logs 로그 그룹을 조회하여 인증서 `<certificate-arn>`과 관련된 ACM 인증서 요청 또는 검증 이벤트를 필터링하고 요청 세부사항을 확인합니다.
8. ACM 인증서 `<certificate-arn>`의 도메인 이름을 조회하여 인증서 요청의 모든 도메인 이름이 유효한지 확인하고 도메인 이름 구성을 확인합니다.
9. CloudTrail 이벤트가 포함된 CloudWatch Logs 로그 그룹을 조회하여 지난 7일 동안 인증서 `<certificate-arn>`과 관련된 ACM 인증서 요청 수정 이벤트를 필터링하고 요청 변경을 확인합니다.

## 진단

1. ACM 인증서 상태 및 도메인 검증 상태(플레이북 2단계)를 분석하여 인증서 발급의 현재 상태를 식별합니다. 상태가 "Pending validation"이면 도메인 검증 프로세스가 완료되지 않은 것입니다. 상태가 "Failed"이면 실패 사유를 조사하여 구체적인 가이드를 확인합니다.

2. ACM 인증서 도메인 검증 레코드(플레이북 5단계)를 검토하여 DNS 또는 이메일 검증 구성을 확인합니다. DNS 검증을 사용하는 경우 DNS 호스팅 영역에 CNAME 레코드가 올바르게 구성되었는지 확인합니다. CNAME 레코드가 누락되었거나 잘못된 경우 DNS 검증을 완료할 수 없습니다. 이메일 검증을 사용하는 경우 도메인 연락처에 검증 이메일이 발송되었는지 확인합니다.

3. DNS 검증 실패의 경우, Route 53 호스팅 영역 구성(플레이북 6단계)을 조사하여 검증 레코드가 존재하고 올바른 형식인지 확인합니다. 검증 레코드의 값이 잘못되었거나 잘못된 호스팅 영역에 있는 경우 ACM 요구사항에 맞게 업데이트합니다.

4. CloudTrail 이벤트가 포함된 CloudWatch Logs(플레이북 7단계 및 9단계)를 분석하여 인증서 요청 패턴과 API 오류를 식별합니다. 로그에서 RequestCertificate API 오류가 확인되면 잘못된 도메인 이름이나 도메인 제한 초과 등 인증서 요청 자체에 문제가 있는 것입니다.

5. 인증서 발급 실패 타임스탬프와 DNS 레코드 수정 타임스탬프를 1시간 이내로 상관관계를 분석합니다. DNS 레코드가 최근에 변경되었거나 삭제된 경우 해당 변경이 진행 중인 검증을 방해했을 수 있습니다. DNS 검증은 검증이 완료될 때까지 레코드가 유지되어야 합니다.

6. 인증서 요청의 여러 도메인에 걸친 인증서 발급 실패 패턴을 비교합니다. 실패가 특정 도메인에 한정되면 해당 도메인의 검증에 문제가 있는 것입니다. 모든 도메인에서 검증이 실패하면 전체 인증서 요청 구성이 잘못되었을 수 있습니다.

7. 인증서가 72시간 이상 보류 상태인 경우 검증 프로세스가 시간 초과되었을 가능성이 높습니다. DNS 검증의 경우 레코드가 여전히 존재하고 접근 가능한지 확인합니다. 이메일 검증의 경우 검증 이메일을 재발송해야 하는지 확인합니다.

지정된 시간 범위 내에서 상관관계가 발견되지 않는 경우: 기간을 30일로 확장하고, DNS 레코드 전파 및 이메일 검증 전달을 포함한 대안적 증거 소스를 검토하고, DNS 레코드 만료나 이메일 검증 시간 초과 같은 점진적 문제를 확인하고, DNS 서비스 가용성이나 이메일 전달 서비스 같은 외부 의존성을 확인하고, 인증서 발급 실패의 과거 패턴을 조사하고, ACM 인증서 요청 속도 제한을 확인하고, ACM 인증서 도메인 제한 제약을 확인합니다. 인증서 발급 실패는 즉각적인 인증서 요청 변경이 아닌 DNS 레코드 전파 지연, 이메일 검증 전달 문제, 인증 기관 서비스 문제, ACM 인증서 요청 속도 제한, ACM 인증서 도메인 제한 제약으로 인해 발생할 수 있습니다.
