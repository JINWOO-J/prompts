# S3 File Upload Extremely Slow - S3 파일 업로드 극도로 느림

## 의미

S3 파일 업로드가 극도로 느린 현상(성능 문제 또는 S3UploadSlow 경보 트리거)은 네트워크 대역폭이 제한되거나, S3 Transfer Acceleration이 활성화되지 않았거나, 멀티파트 업로드 구성이 최적이 아니거나, 객체 크기가 최적 임계값을 초과하거나, 리전 네트워크 지연이 높거나, S3 요청 스로틀링이 업로드 성능에 영향을 미칠 때 발생합니다.
 S3 업로드 성능이 저하되고, 파일 전송 시간이 크게 증가하며, 애플리케이션 업로드 작업이 타임아웃됩니다. 이는 스토리지 및 데이터 전송 계층에 영향을 미치며 사용자 경험에 영향을 줍니다. 일반적으로 네트워크 문제, 전송 구성 문제 또는 스로틀링이 원인이며, Transfer Acceleration과 함께 S3를 사용하는 경우 구성이 다를 수 있고 애플리케이션에서 업로드 성능 문제가 발생할 수 있습니다.

## 영향

S3 업로드 성능이 저하됩니다. 파일 전송 시간이 크게 증가합니다. 애플리케이션 업로드 작업이 타임아웃됩니다. 사용자 경험이 영향을 받습니다. 업로드 처리량이 감소합니다. 대용량 파일 전송이 실패합니다. 업로드 작업이 과도한 시간을 소비합니다. 애플리케이션 성능이 심각하게 영향을 받습니다. S3UploadSlow 경보가 발생할 수 있으며, Transfer Acceleration과 함께 S3를 사용하는 경우 구성이 다를 수 있습니다. 느린 업로드로 인해 애플리케이션에서 오류나 성능 저하가 발생할 수 있으며, 사용자 대면 업로드 기능을 사용할 수 없게 될 수 있습니다.

## 플레이북

1. S3 버킷 `<bucket-name>`이 존재하고 리전 `<region>`의 S3 AWS 서비스 상태가 정상인지 확인합니다.
2. S3 버킷 `<bucket-name>`의 CloudWatch 지표(BytesUploaded, PutRequests)를 최근 1시간 동안 조회하여 업로드 성능 패턴을 파악하고, 업로드 처리량을 분석합니다.
3. 리전 `<region>`의 S3 버킷 `<bucket-name>`을 조회하여 Transfer Acceleration 구성, 요청 지표 구성, 버킷 위치를 점검하고, Transfer Acceleration 상태를 검증합니다.
4. S3 서버 접근 로그가 포함된 CloudWatch Logs 로그 그룹에서 버킷 `<bucket-name>` 관련 느린 업로드 패턴이나 타임아웃 이벤트를 필터링하여 업로드 지속 시간 패턴을 확인합니다.
5. 네트워크 지연 및 대역폭 사용률을 포함한 네트워크 성능 CloudWatch 지표를 최근 1시간 동안 조회하여 네트워크 제약을 파악하고, 네트워크 지표를 분석합니다.
6. 버킷 `<bucket-name>`의 S3 업로드 작업을 나열하고 업로드 지속 시간 패턴과 객체 크기 분포를 분석하여 대용량 객체가 성능에 영향을 미치는지 확인합니다.
7. S3 버킷 `<bucket-name>`의 CloudWatch 지표(ThrottledRequests, 가용한 경우)를 조회하여 요청 스로틀링이 업로드에 영향을 미치는지 확인하고, 스로틀링 패턴을 점검합니다.
8. S3 버킷 `<bucket-name>`의 요청 지표 구성을 조회하여 요청 지표가 활성화되어 있는지 확인하고, 지표가 업로드 성능 인사이트를 제공하는지 점검합니다.
9. CloudTrail 이벤트가 포함된 CloudWatch Logs 로그 그룹에서 최근 24시간 이내 버킷 `<bucket-name>` 관련 S3 Transfer Acceleration 또는 요청 지표 구성 변경 이벤트를 필터링하여 구성 변경 사항을 확인합니다.

## 진단

1. S3 버킷 업로드의 CloudWatch 지표(플레이북 2단계)를 분석하여 최근 1시간 동안의 BytesUploaded 및 PutRequests를 포함한 업로드 처리량 패턴을 파악합니다. 처리량이 갑자기 감소했다면 타임스탬프를 네트워크 또는 구성 변경과 상관 분석합니다. 처리량이 지속적으로 낮았다면 지속적인 구성 문제일 가능성이 높습니다.

2. 네트워크 성능 지표(플레이북 5단계)를 검토하여 네트워크 지연 및 대역폭 사용률을 확인합니다. 네트워크 지연이 높거나 대역폭이 제한되면 업로드 성능에 직접 영향을 미칩니다. 네트워크 지표가 정상이면 S3 구성 또는 클라이언트 측 문제일 가능성이 높습니다.

3. S3 서버 접근 로그(플레이북 4단계)를 확인하여 업로드 지속 시간 패턴을 분석하고 느린 업로드 이벤트를 파악합니다. 로그에서 타임아웃이나 재시도와 같은 특정 패턴이 표시되면 네트워크 불안정이 업로드 성능 문제를 유발할 수 있습니다.

4. S3 Transfer Acceleration 구성(플레이북 3단계)을 확인하여 버킷에 가속이 활성화되어 있는지 판단합니다. 버킷 리전에서 먼 지리적 위치에서 업로드하는 경우 Transfer Acceleration을 활성화하면 업로드 속도를 크게 향상시킬 수 있습니다.

5. 객체 크기 분포(플레이북 6단계)를 분석하여 대용량 객체가 업로드되고 있는지 판단합니다. 100MB보다 큰 객체의 경우 멀티파트 업로드를 사용해야 합니다. 대용량 객체가 단일 요청으로 업로드되면 업로드 성능이 저하되고 타임아웃 가능성이 높아집니다.

6. S3 요청 스로틀링(플레이북 7단계)을 확인하여 가용한 경우 ThrottledRequests 지표를 점검합니다. 버킷이 매우 높은 요청 볼륨(특히 공통 접두사)을 받고 있으면 S3가 요청을 스로틀링하여 속도 저하를 유발할 수 있습니다.

7. S3 요청 지표 구성(플레이북 8단계)을 검토하여 성능 분석을 위한 상세 지표가 활성화되어 있는지 확인합니다. 요청 지표가 구성되지 않은 경우 업로드 성능에 대한 더 나은 가시성을 위해 활성화합니다.

8. CloudTrail 이벤트(플레이북 9단계)와 업로드 속도 저하 타임스탬프를 30분 이내로 상관 분석하여 성능에 영향을 미쳤을 수 있는 Transfer Acceleration 또는 버킷 구성 변경을 파악합니다.

9. 최근 4시간 동안의 업로드 성능 추세를 비교합니다. 느린 성능이 지속적이면 Transfer Acceleration이나 멀티파트 설정과 같은 구성 문제에 집중합니다. 느린 성능이 간헐적이면 네트워크 문제나 요청 스로틀링에 집중합니다.

상관관계를 찾을 수 없는 경우: 기간을 24시간으로 확장하고, 멀티파트 업로드 구성 및 클라이언트 측 업로드 설정을 포함한 대안적 증거 소스를 검토하고, 네트워크 혼잡이나 리전 용량 제약과 같은 점진적 문제를 확인하고, 인터넷 연결이나 VPN 터널 성능과 같은 외부 종속성을 검증하고, 업로드 성능의 과거 패턴을 점검합니다.
