# Socratic Method — 소크라틱 질문법

## 개념

답을 직접 제시하지 않고, 체계적인 질문을 통해 상대방(또는 자기 자신)이
스스로 깊은 이해에 도달하도록 유도하는 기법. 소크라테스의 대화법에서 유래.
Richard Paul의 6가지 소크라틱 질문 유형이 체계적 프레임워크로 널리 사용된다.

## 핵심 원리

- 능동적 학습: 답을 받는 것보다 질문을 통해 스스로 발견하면 이해가 깊어짐
- 가정 노출: 질문을 통해 숨겨진 가정과 편향이 드러남
- 비판적 사고 훈련: 근거 없는 주장을 걸러내는 능력 향상
- Richard Paul의 6유형: 명확화, 가정 탐색, 근거 질문, 관점 전환, 결과 탐색, 메타 질문

## 프롬프트 템플릿

```
다음 주제에 대해 소크라틱 질문법으로 탐구하세요.

**주제:** [탐구할 주제]

**1단계 — 명확화 질문 (Clarification):**
- 정확히 무엇을 의미하는가?
- 핵심 개념을 한 문장으로 정의할 수 있는가?
- 구체적인 예를 들 수 있는가?

**2단계 — 가정 탐색 (Probing Assumptions):**
- 어떤 가정을 하고 있는가?
- 그 가정이 틀리다면 결론이 어떻게 바뀌는가?
- 이 가정은 항상 성립하는가?

**3단계 — 근거 질문 (Probing Reasons/Evidence):**
- 그렇게 생각하는 근거는 무엇인가?
- 이 근거는 얼마나 신뢰할 수 있는가?
- 반대 증거는 없는가?

**4단계 — 관점 전환 (Questioning Viewpoints):**
- 다른 관점에서 보면 어떤가?
- [반대 입장]은 어떻게 반박할까?
- 이해관계자별로 어떻게 다르게 보는가?

**5단계 — 결과 탐색 (Probing Implications):**
- 이 결론의 함의는 무엇인가?
- 어떤 결과가 따르는가?
- 최악의 경우는?

**6단계 — 메타 질문 (Questions about the Question):**
- 왜 이 질문이 중요한가?
- 우리가 놓치고 있는 것은 무엇인가?
- 더 좋은 질문은 무엇인가?
```

## 실전 예시

**시나리오: "마이크로서비스로 전환해야 하는가?"**

명확화: "마이크로서비스"가 정확히 무엇을 의미하는가? 독립 배포? 독립 DB?
가정 탐색: "모놀리스는 확장이 안 된다"는 가정이 사실인가? → Shopify는 모놀리스로 수십억 트래픽 처리
근거: 전환이 필요하다는 근거는? → "팀이 커져서" → 팀 크기와 아키텍처의 상관관계는?
관점 전환: 주니어 개발자 관점에서는? → 복잡도 증가로 온보딩 시간 3배
결과: 전환하면? → 운영 복잡도 증가, 분산 트랜잭션 문제, 네트워크 레이턴시
메타: 진짜 문제는 아키텍처인가, 팀 구조인가?

## 변형 및 조합

- **Socratic + Devil's Advocate**: 질문으로 반론을 유도
- **Socratic + First Principles**: 질문으로 근본 가정까지 파고들기
- **Socratic + CoT**: 각 질문의 답을 단계별로 추론

## 주의사항

- 시간이 많이 소요됨 — 긴급한 의사결정에는 부적합
- 질문이 너무 추상적이면 실용적 결론에 도달하기 어려움
- 상대방(또는 LLM)이 질문의 의도를 오해할 수 있음
- 답을 이미 알고 있는 경우에는 비효율적

## 적합한 상황

깊은 분석, 의사결정 검증, 팀 토론, 요구사항 도출, 코드 리뷰, 아키텍처 결정, 교육
