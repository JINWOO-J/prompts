# MECE Framework — 상호배제·전체포괄 분류

## 개념

Mutually Exclusive, Collectively Exhaustive.
모든 항목이 겹치지 않으면서(ME) 전체를 빠짐없이 포괄(CE)하도록 분류하는 기법.
맥킨지 컨설팅의 Barbara Minto가 체계화. 문제 구조화의 황금 표준.

## 핵심 원리

- 누락 방지(CE): 모든 가능성을 포괄하여 빠뜨리는 것이 없음
- 중복 방지(ME): 카테고리 간 겹침이 없어 이중 계산/분석 방지
- 커뮤니케이션 명확성: MECE 구조는 듣는 사람이 이해하기 쉬움
- 피라미드 원칙의 기반: 논리적 문서/프레젠테이션 구조의 핵심

## 프롬프트 템플릿

```
다음 주제를 MECE 원칙으로 분류하세요.

**주제:** [분류할 대상]

**MECE 분류:**

| # | 카테고리 | 포함 항목 | 제외 항목 (ME 검증) |
|---|----------|----------|-------------------|
| 1 | [카테고리명] | [...] | [...] |
| 2 | [카테고리명] | [...] | [...] |
| 3 | [카테고리명] | [...] | [...] |
| 4 | [카테고리명] | [...] | [...] |

**CE 검증 (전체 포괄):**
- 위 카테고리로 전체가 포괄되는가? [예/아니오]
- 누락된 항목: [있다면 기재, 없으면 "없음"]
- 포괄률: [추정 %]

**ME 검증 (상호 배제):**
- 카테고리 간 겹치는 항목: [있다면 기재, 없으면 "없음"]
- 겹침 해소 방법: [재분류 방안]

**최종 MECE 구조:**
[검증 후 확정된 분류]
```

## 실전 예시

**시나리오: "API 응답 지연 원인 분류"**

MECE 분류:
1. 네트워크 계층: DNS, TCP, TLS, 라우팅
2. 애플리케이션 계층: 코드 로직, 메모리, GC, 스레드
3. 데이터 계층: DB 쿼리, 캐시 미스, 커넥션 풀
4. 외부 의존성: 서드파티 API, CDN, 외부 서비스

CE 검증: 모든 요청 경로가 포괄됨 ✅
ME 검증: "DB 커넥션 풀"이 네트워크와 데이터 양쪽에 걸릴 수 있음 → 데이터 계층으로 확정

## 변형 및 조합

- **MECE + 5 Whys**: 각 카테고리 내에서 5 Whys 적용
- **MECE + Fishbone**: MECE 카테고리를 Fishbone의 뼈대로 사용
- **MECE + Decomposition**: 각 카테고리를 하위 MECE로 재분해
- **MECE + Prioritization**: MECE 분류 후 우선순위 매기기

## 주의사항

- 완벽한 MECE는 현실에서 달성하기 어려움 (근사적 MECE로 충분)
- 과도한 분류는 실용성을 해침 (3-7개 카테고리가 적정)
- 분류 기준(축)의 선택이 결과를 좌우 — 여러 축을 시도해볼 것
- 동적으로 변하는 대상에는 MECE가 깨질 수 있음

## 적합한 상황

문제 구조화, 원인 분류, 요구사항 정리, 전략 수립, 보고서 구조화, 이슈 트리아지
