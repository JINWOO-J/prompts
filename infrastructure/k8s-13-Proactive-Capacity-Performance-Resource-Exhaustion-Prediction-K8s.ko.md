# Resource Exhaustion Prediction (리소스 고갈 예측)

## 의미

리소스 고갈 예측 알림은 리소스 용량 고갈을 정확히 예측할 수 없거나 리소스 고갈 경고가 생성되지 않는 상황(ResourceExhaustionPredictionUnavailable 또는 CapacityExhaustionWarningFailed 같은 알림 발생)을 나타냅니다. 리소스 고갈 예측 모델 실패, 리소스 용량 추세 분석 불가, 리소스 고갈 경고 미생성, 리소스 쿼터 사용률 예측 미가용, 또는 리소스 고갈 타임라인 계산 불가 등이 원인입니다.
 리소스 고갈 예측이 미가용하고, 리소스 용량 추세가 예측 불가능한 패턴을 보이며, 리소스 고갈 경고가 생성되지 않고, 리소스 고갈 타임라인 계산이 실패합니다. 이는 용량 계획 계층과 리소스 관리에 영향을 미치며, 주로 불충분한 리소스 사용 데이터, 고갈 예측 모델 실패, 리소스 추세 분석 문제, 또는 고갈 경고 자동화 실패가 원인입니다. 리소스 고갈이 컨테이너 워크로드에 영향을 미치는 경우, 컨테이너 리소스가 예기치 않게 고갈되고 애플리케이션이 용량 실패를 겪을 수 있습니다.

## 영향

ResourceExhaustionPredictionUnavailable 알림 발생, CapacityExhaustionWarningFailed 알림 발생, 리소스 고갈 예측 불가, 리소스 고갈 경고 미생성, 용량 실패가 예기치 않게 발생 가능, 리소스 프로비저닝 지연 가능. 리소스 고갈 예측이 미가용합니다. 리소스 고갈이 컨테이너 워크로드에 영향을 미치는 경우, 컨테이너 리소스가 예기치 않게 고갈되고, Pod 리소스가 소진되며, 컨테이너 애플리케이션이 용량 실패를 겪을 수 있습니다. 애플리케이션이 리소스 고갈이나 예기치 않은 용량 제약을 경험할 수 있습니다.

## 플레이북

1. 모든 노드를 describe하여 노드 용량, 할당 가능 리소스, 현재 사용률을 점검하고 잠재적 리소스 고갈 지점을 식별합니다.
2. 모든 namespace에서 최근 이벤트를 타임스탬프 순으로 조회하여 리소스 압박, OOMKilled Pod, 용량 제약 관련 이벤트를 확인합니다.
3. namespace <namespace>에서 리소스 쿼터를 YAML 출력으로 조회하여 고갈 예측 분석을 위한 현재 리소스 쿼터 사용량과 제한을 확인합니다.
4. Pod 리소스 사용률의 Prometheus 메트릭(CPU 사용률, 메모리 사용률, 스토리지 사용률)을 최근 90일 동안 조회하여 리소스 고갈 추세를 분석합니다.
5. 리소스 쿼터 사용률의 Prometheus 메트릭(쿼터 사용 비율, 쿼터 제한 근접도)을 최근 90일 동안 조회하여 쿼터 고갈 접근을 식별합니다.
6. 리소스 모니터링 Pod의 로그를 조회하고 최근 90일 내 리소스 소진이나 용량 고갈을 나타내는 패턴을 필터링합니다.
7. 노드 용량의 Prometheus 메트릭(가용 용량, 용량 사용률)을 최근 90일 동안 조회하여 용량 고갈 추세를 식별합니다.
8. 리소스 사용률 추세 데이터와 리소스 고갈 예측 모델 출력을 비교하여 예측이 접근하는 고갈을 식별하는지 확인하고, Prometheus 메트릭을 보조 증거로 활용합니다.

## 진단

1. 1단계와 3단계의 노드 용량 및 리소스 쿼터 데이터를 검토합니다. 현재 사용률이 제한에 근접하면 고갈 예측이 제한 도달 시점을 정확히 예측하고 있는지 확인합니다.

2. 4~5단계의 90일 사용률 추세를 분석합니다. 추세가 일관된 증가를 보이면 예측이 고갈 타임라인을 식별해야 합니다. 추세가 불규칙하면 예측이 신뢰할 수 없으며 반응형 모니터링을 강조해야 합니다.

3. 6단계의 리소스 모니터링 로그에서 소진 패턴이 있으면 어떤 리소스(CPU, 메모리, 스토리지, Pod)가 고갈에 접근하고 있는지 식별하고 해당 리소스의 용량 확장을 우선합니다.

4. 7단계의 용량 추세 데이터를 검토합니다. 용량 사용률이 용량 증가보다 빠르게 증가하면 고갈이 예상됩니다. 용량 증가가 사용률 증가를 앞서면 현재 용량 계획이 적절합니다.

5. 8단계의 예측 모델 출력이 실제 사용률과 크게 다르면 예측 모델을 현재 데이터 패턴으로 재보정해야 합니다.

분석이 결론에 이르지 못하면: 2단계의 이벤트에서 이미 발생하고 있는 고갈을 나타내는 리소스 압박이나 OOMKilled 이벤트를 확인합니다. 고갈 위험이 특정 리소스 유형에 집중되어 있는지(대상 확장 시사) 또는 분산되어 있는지(전체 용량 계획 필요 시사) 판단합니다. 예측 모델이 정확한 예측을 위한 충분한 이력 데이터를 보유하고 있는지 확인합니다.
