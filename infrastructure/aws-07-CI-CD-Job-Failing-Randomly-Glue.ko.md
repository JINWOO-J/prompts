# AWS Glue 작업 무작위 실패

## 의미

AWS Glue 작업이 무작위로 실패(작업 실행 실패 또는 GlueJobFailure 알람 트리거)하는 이유는 작업 스크립트에 간헐적 오류가 있거나, 작업 리소스가 부족하거나, 데이터 소스 연결이 불안정하거나, 작업 타임아웃이 발생하거나, Glue 서비스가 작업 실행 중 오류를 만나거나, Glue 작업 DPU 할당이 부족하기 때문입니다. Glue 작업이 무작위로 실패하고, 데이터 처리 워크플로우가 중단되며, 작업 실행 신뢰성이 손상됩니다. 이는 데이터 처리 및 ETL 계층에 영향을 미치며 데이터 파이프라인을 방해합니다. 일반적으로 스크립트 문제, 리소스 제약 또는 연결 문제로 인해 발생합니다. 다른 작업 유형(Spark, Python shell)으로 Glue를 사용하는 경우 실패 패턴이 다를 수 있으며 애플리케이션에서 간헐적 작업 실패가 발생할 수 있습니다.

## 영향

Glue 작업 무작위 실패; 데이터 처리 워크플로우 중단; 작업 실행 신뢰성 손상; 작업 실패 예측 불가; 데이터 처리 자동화 무효; 작업 재시도 실패; ETL 프로세스 방해; 데이터 파이프라인 신뢰성 영향. GlueJobFailure 알람이 발생할 수 있으며, 다른 작업 유형(Spark, Python shell)으로 Glue를 사용하는 경우 실패 패턴이 다를 수 있고, 실패한 데이터 처리로 인해 애플리케이션에서 오류 또는 성능 저하가 발생할 수 있으며, ETL 파이프라인이 불안정할 수 있습니다.

## 플레이북

1. Glue 작업 `<job-name>`이 존재하고 리전 `<region>`에서 Glue의 AWS 서비스 상태가 정상인지 확인합니다.
2. 리전 `<region>`에서 Glue 작업 `<job-name>`을 조회하고 작업 구성, 스크립트 위치, 리소스 할당, 타임아웃 설정, 작업 실행 이력을 검사하여 작업 구성을 확인합니다.
3. 로그 그룹 `/aws-glue/jobs/<job-name>`의 CloudWatch Logs를 쿼리하고 작업 실패 패턴, 오류 메시지 또는 타임아웃 이벤트를 필터링하여 오류 메시지 세부 정보를 포함합니다.
4. 지난 7일 동안 Glue 작업 `<job-name>`의 JobRunsFailed 및 JobRunsSucceeded를 포함한 CloudWatch 메트릭을 조회하여 실패 패턴을 식별하고 실패 빈도를 분석합니다.
5. 작업 `<job-name>`의 Glue 작업 실행을 나열하고 작업 실행 상태, 실패 이유, 실행 기간, 리소스 활용도를 확인하여 작업 실행 패턴을 분석합니다.
6. Glue 데이터 소스 연결 로그가 포함된 CloudWatch Logs 로그 그룹을 쿼리하고 연결 오류 또는 데이터 소스 접근 실패를 필터링하여 연결 오류 세부 정보를 포함합니다.
7. Glue 작업 `<job-name>` DPU 할당을 조회하고 DPU 구성을 확인하여 리소스 제약이 실패를 유발하는지 확인합니다.
8. Glue 작업 `<job-name>`의 JobRunExecutionTime을 포함한 CloudWatch 메트릭을 조회하고 작업 실행 기간 패턴을 확인하여 타임아웃이 실패를 유발하는지 확인합니다.
9. CloudTrail 이벤트가 포함된 CloudWatch Logs 로그 그룹을 쿼리하고 지난 7일 이내 작업 `<job-name>`과 관련된 Glue 작업 구성 또는 스크립트 수정 이벤트를 필터링하여 구성 변경을 확인합니다.

## 진단

1. **3단계의 CloudWatch Logs 분석**: 오류 패턴과 실패 메시지에 대한 Glue 작업 로그를 검토합니다. CloudWatch Logs가 "OutOfMemoryError", "heap space" 또는 DPU 관련 오류를 나타내면 리소스 부족이 실패를 유발하고 있습니다(7단계 데이터로 진행). 로그가 "Connection refused", "timeout" 또는 데이터 소스 접근 오류를 표시하면 연결 문제가 원인입니다(6단계 데이터로 진행). 로그가 스크립트 예외를 표시하면 2단계로 계속합니다.

2. **5단계의 작업 실행 패턴 평가**: 5단계의 작업 실행이 특정 시간, 데이터 볼륨 또는 일관된 기간 실행 후 실패를 표시하면 패턴이 리소스 고갈 또는 타임아웃 문제를 시사합니다. 실패가 2단계의 타임아웃 설정에 접근하는 실행 기간과 상관되면 작업 타임아웃이 원인입니다. 실패가 패턴 없이 진정으로 무작위인 경우 3단계로 계속합니다.

3. **6단계의 데이터 소스 연결 확인**: 6단계의 CloudWatch Logs가 작업 실패 5분 이내에 데이터 소스 연결 오류를 표시하면 불안정한 데이터 소스 연결이 무작위 실패를 유발하고 있습니다. JDBC 연결 안정성, S3 접근 패턴, 데이터 소스에 대한 네트워크 연결을 확인합니다. 연결이 안정적인 것으로 보이면 4단계로 계속합니다.

4. **7단계 및 8단계의 DPU 할당 및 메트릭 검토**: 7단계의 DPU 할당이 처리 중인 데이터 볼륨에 부족하면 리소스 제약이 부하 시 간헐적 실패를 유발합니다. 4단계의 CloudWatch 메트릭이 작업 실행 시간 스파이크와 실패 간의 상관관계를 표시하면 리소스 경합이 문제입니다. 성공한 실행과 실패한 실행을 비교합니다 - 실패한 실행이 더 큰 데이터셋을 처리한 경우 스케일링 문제가 존재합니다.

5. **9단계의 구성 변경과 상관관계**: 9단계의 CloudTrail 이벤트가 실패가 시작되기 30분 이내에 작업 스크립트 또는 구성 수정을 표시하면 최근 변경이 불안정성을 도입했습니다. 오류 처리, 연결 관리 또는 리소스 할당의 잠재적 문제를 식별하기 위해 특정 변경을 검토합니다.

**상관관계를 찾을 수 없는 경우**: 5단계의 작업 실행 데이터를 사용하여 분석을 90일로 확장합니다. 실패가 진정으로 무작위인 경우 Glue 서비스 상태와 데이터 소스 성능 저하를 조사합니다. Spark 작업의 경우 데이터 스큐 또는 파티션 문제를 확인합니다. Python shell 작업의 경우 의존성 호환성을 확인합니다. 작업 재시도 동작과 일시적 실패가 적절히 처리되고 있는지 검토합니다.
