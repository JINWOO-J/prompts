# KubePersistentVolumeFillingUp

## 의미

PersistentVolume이 용량 한계에 근접하고 있습니다(PersistentVolume 용량 관련 알림 트리거). 디스크 사용량이 증가하여 곧 가용 스토리지 공간이 고갈될 수 있습니다. 클러스터 대시보드에서 PersistentVolume의 높은 디스크 사용량이 표시되며, 스토리지 사용량 메트릭이 증가 추세를 나타내고, 애플리케이션 로그에 스토리지 관련 오류가 표시될 수 있습니다. 이는 스토리지 플레인에 영향을 미치며 데이터 증가, 보존 정책 부재, 또는 불충분한 스토리지 용량으로 인해 애플리케이션이 데이터를 쓸 수 없게 됩니다.

## 영향

PersistentVolume 용량 알림 발생, 서비스 저하, 애플리케이션이 읽기 전용 모드로 전환될 수 있음, 데이터 쓰기 실패 가능, 스토리지 고갈로 새 데이터 저장 불가, 애플리케이션 충돌 또는 사용 불가, 스토리지가 완전히 차면 데이터 손실 위험, 볼륨 사용 불가 가능. PersistentVolume이 용량 한계에 근접한 높은 디스크 사용량을 표시하며, 스토리지 사용량 메트릭이 증가 추세를 나타내고, 스토리지가 차면 애플리케이션이 읽기 전용 모드로 전환되거나 충돌할 수 있습니다.

## 플레이북

1. PersistentVolume <pv-name>을 describe하여 볼륨 상태, 용량, 사용 공간, reclaim policy, 스토리지 문제를 나타내는 조건을 확인합니다.

2. PersistentVolume <pv-name>의 이벤트를 타임스탬프 순으로 조회하여 용량 경고 또는 스토리지 백엔드 문제를 식별합니다.

3. namespace <namespace>에서 PersistentVolumeClaim <pvc-name>을 describe하여 PVC 상태, 요청 용량, 실제 사용량을 확인합니다.

4. PersistentVolume <pv-name>의 지난 7일간 스토리지 사용량 추세 메트릭을 조회하여 증가 패턴을 식별합니다.

5. 스토리지 사용량에 영향을 줄 수 있는 애플리케이션 데이터 보존 정책과 구성을 확인합니다.

6. 스토리지 공간을 소비할 수 있는 스냅샷 또는 백업 구성을 확인합니다.

7. PersistentVolume <pv-name>을 사용하는 namespace <namespace>의 Pod <pod-name> 로그를 조회하여 데이터 증가 패턴을 식별합니다.

8. 노드에 SSH 접속하여 일반적인 스토리지 소비자의 디스크 사용량을 확인합니다: /var/log의 누적 로그, 캐시된 컨테이너 이미지, 고아 Pod 데이터.

## 진단

1. 플레이북 2단계의 PV 이벤트를 분석하여 용량 경고 또는 스토리지 오류를 식별합니다. 용량 임계값 초과를 보여주는 이벤트는 용량 부족이 시작된 시점을 나타냅니다. 최근 경고가 표시되면 타임스탬프를 애플리케이션 활동과 상관 분석합니다.

2. 최근 용량 경고가 있으면 플레이북 4단계의 스토리지 사용량 추세를 조사하여 증가 패턴을 파악합니다:
   - 급격한 증가 (수 시간 내) - 잘못된 구성, 데이터 누수, 예상치 못한 대량 데이터 가져오기 가능성
   - 점진적 증가 (수일/수주에 걸쳐) - 용량 계획을 초과하는 정상적인 데이터 증가
   - 단계적 증가 - 배포 또는 스케일링 이벤트와 상관 분석

3. 증가 패턴이 급격한 경우, 플레이북 7단계의 애플리케이션 로그에서 다음을 확인합니다:
   - 과도한 로깅 또는 디버그 모드 활성화
   - 대량 데이터 작업 또는 가져오기
   - 임시 데이터를 남기는 실패한 정리 작업
   - 데이터 누적을 유발하는 재시도 폭주와 함께 발생하는 애플리케이션 오류

4. 증가 패턴이 점진적인 경우, 플레이북 5단계의 데이터 보존 정책을 조사합니다. 보존 정책이 없거나 "영구 보관"으로 설정된 경우 데이터 누적은 예상된 결과입니다. 현재 보존 설정을 실제 스토리지 사용량과 비교하여 정책이 적용되고 있는지 확인합니다.

5. 플레이북 6단계에서 스냅샷 또는 백업 구성이 존재하면, 스냅샷이 동일 볼륨의 공간을 소비하는지 확인합니다. 일부 스토리지 백엔드는 스냅샷을 볼륨 용량에 포함합니다.

6. PV가 노드에 있는 경우, 플레이북 8단계에서 노드 수준 디스크 소비자를 확인합니다. /var/log 또는 컨테이너 이미지가 주요 소비자라면, 애플리케이션 데이터 증가가 아닌 노드 디스크 압력 문제입니다.

7. 단일 원인이 식별되지 않으면, 현재 채움 속도를 과거 데이터와 비교하여 볼륨이 가득 찰 시점을 예측합니다. 이를 통해 즉각적인 정리 또는 용량 확장의 우선순위를 결정합니다.

**지정된 시간 범위 내에서 상관관계를 찾을 수 없는 경우**: 용량 계획 분석을 위해 시간 범위를 90일로 확장하고, 애플리케이션 데이터 보존 요구사항을 검토하고, 데이터 누수 또는 잘못된 구성을 확인하고, 스냅샷 및 백업 보존 정책을 검증하고, 과거 스토리지 증가 패턴을 조사합니다. PersistentVolume 용량 부족은 즉각적인 운영 변경이 아닌 정상적인 데이터 증가, 부적절한 보존 정책, 불충분한 스토리지 용량으로 인해 발생할 수 있습니다.
