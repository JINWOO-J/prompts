# Devil's Advocate — 반론 제기 기법

## 개념

의도적으로 반대 입장을 취하여 주장의 약점을 찾아내는 기법.
13세기 가톨릭 시성 절차에서 유래 (Advocatus Diaboli).
집단사고(groupthink)를 방지하고 의사결정의 견고성을 높인다.

## 핵심 원리

- 확증 편향 극복: 자신의 주장에 유리한 증거만 찾는 경향을 의식적으로 깨뜨림
- 약점 사전 발견: 실행 전에 약점을 찾아 보완할 수 있음
- 의사결정 품질 향상: 반론을 견딘 주장은 더 견고함
- 심리적 안전: "역할"로서 반론하므로 개인 공격이 아닌 건설적 비판

## 프롬프트 템플릿

```
**원래 주장/제안:** [주장 설명]

**Devil's Advocate 분석:**

**1. 핵심 약점 (Top 3):**
1. [약점]: [왜 문제인지, 구체적 시나리오]
2. [약점]: [왜 문제인지, 구체적 시나리오]
3. [약점]: [왜 문제인지, 구체적 시나리오]

**2. 숨겨진 가정:**
- [가정 1]: 이것이 틀리면? → [결과]
- [가정 2]: 이것이 틀리면? → [결과]

**3. 반대 입장에서의 대안:**
- 대안 A: [설명] — 장점: [...], 단점: [...]
- 대안 B: [설명] — 장점: [...], 단점: [...]

**4. 실패 시나리오:**
- 최악의 경우: [시나리오와 영향]
- 가장 가능성 높은 실패: [시나리오와 영향]

**5. 최종 판단:**
원래 주장을 유지/수정/폐기해야 하는가?
- 판단: [유지/수정/폐기]
- 근거: [...]
- 수정한다면: [구체적 보완 방안]
```

## 실전 예시

**시나리오: "Redis를 세션 스토어로 사용하자"**

약점 1: Redis 장애 시 전체 사용자 로그아웃 → 단일 장애점
약점 2: 메모리 비용이 DB 대비 10배 → 사용자 증가 시 비용 폭증
약점 3: 세션 데이터 영속성 미보장 → 재시작 시 데이터 유실

숨겨진 가정: "세션은 빠른 읽기가 필요하다" → 실제로 세션 조회가 병목인가?

대안: JWT 토큰 (서버 상태 불필요) / DB + 캐시 레이어

판단: 수정 — Redis Sentinel + AOF 영속성 + JWT 하이브리드

## 변형 및 조합

- **Devil's Advocate + Six Hats**: Black Hat을 Devil's Advocate로 강화
- **Devil's Advocate + Red Team**: 보안 관점에서 공격 시나리오 추가
- **Devil's Advocate + Pre-mortem**: 실패 시나리오를 더 체계적으로

## 주의사항

- 과도한 반론은 의사결정을 마비시킬 수 있음 (analysis paralysis)
- 반론이 건설적이어야 함 — 단순 부정이 아닌 대안 제시
- 팀에서 사용 시 "역할"임을 명확히 해야 개인 갈등 방지
- 모든 결정에 적용하면 비효율적 — 중요한 결정에만 사용

## 적합한 상황

의사결정 검증, 아키텍처 리뷰, 제안서 검토, 리스크 평가, 투자 결정
