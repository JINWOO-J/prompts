# Hypothesis-Driven Analysis — 가설 기반 분석

## 개념

과학적 방법론을 문제 해결에 적용. 가설을 세우고,
검증 방법을 설계하고, 데이터로 확인/반증하는 체계적 접근.
Karl Popper의 반증주의(Falsificationism)가 이론적 기반.
"가설이 틀렸음을 증명하려고 노력하라. 살아남은 가설이 가장 유력하다." 

## 핵심 원리

- 효율적 탐색: 무작위 조사 대신 가설 기반으로 탐색 범위를 좁힘
- 반증 가능성: 좋은 가설은 "틀렸음을 증명할 수 있는" 가설
- 편향 감소: 확증 편향 대신 반증을 시도하면 객관성 향상
- 우선순위: 가설의 영향도와 검증 용이성으로 순서 결정
- 맥킨지 방법론: "Day 1 Answer" — 첫날부터 가설을 세우고 검증

## 프롬프트 템플릿

```
**관찰된 현상:** [문제/증상 설명]

**가설 수립:**

| # | 가설 | 예상 근거 | 반증 조건 | 검증 방법 | 비용/시간 | 우선순위 |
|---|------|----------|----------|----------|----------|---------|
| H1 | [가설 1] | [...] | [이것이 관찰되면 기각] | [...] | [...] | [1/2/3] |
| H2 | [가설 2] | [...] | [이것이 관찰되면 기각] | [...] | [...] | [1/2/3] |
| H3 | [가설 3] | [...] | [이것이 관찰되면 기각] | [...] | [...] | [1/2/3] |

**검증 실행 (우선순위 순):**

**H1 검증:**
- 실행: [구체적 검증 행동]
- 예상 결과 (가설이 맞다면): [...]
- 실제 결과: [데이터/관찰]
- 판정: ✅ 확인 / ❌ 반증 / ⚠️ 불확실

**H2 검증:** (H1이 반증된 경우)
- 실행: [...]
- 실제 결과: [...]
- 판정: [...]

**결론:**
- 확인된 가설: [...]
- 반증된 가설: [...]
- 남은 불확실성: [...]
- 다음 행동: [확인된 가설 기반 조치]
```

## 실전 예시

**시나리오: "API 응답 시간이 간헐적으로 10초 초과"**

H1: DB 슬로우 쿼리 (근거: 최근 데이터 증가) → 검증: slow query log 확인
H2: GC 일시 정지 (근거: Java 앱, 힙 사용률 높음) → 검증: GC 로그 분석
H3: 외부 API 타임아웃 (근거: 서드파티 의존) → 검증: 외부 호출 레이턴시 메트릭

H1 검증: slow query log에 10초 이상 쿼리 없음 → ❌ 반증
H2 검증: GC 로그에 Full GC 8초 발생, 시간대 일치 → ✅ 확인
→ 결론: G1GC로 전환 + 힙 사이즈 조정

## 변형 및 조합

- **Hypothesis + CoT**: 각 가설의 검증 과정을 CoT로 상세 추론
- **Hypothesis + MECE**: 가설을 MECE로 분류하여 누락 방지
- **Hypothesis + Bayesian**: 사전 확률을 부여하고 증거에 따라 업데이트
- **Hypothesis + A/B Test**: 가설을 A/B 테스트로 검증

## 주의사항

- 가설이 편향되면 검증도 편향됨 — 다양한 가설 생성이 중요
- 확증 편향: 가설을 "증명"하려 하지 말고 "반증"하려 노력해야 함
- 모든 가설이 반증되면 새로운 가설이 필요 — 열린 마음 유지
- 검증 비용이 높은 가설은 후순위로 미루되, 영향도가 크면 예외
