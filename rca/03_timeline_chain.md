---
기법: 프롬프트 체이닝 (Prompt Chaining) — 4단계 파이프라인
난이도: 중급
적합 상황: 멀티 서비스/시스템 장애, 로그 소스가 다양하고 타임라인 재구성이 필요한 경우
필수 입력: 각 시스템 로그 (최소 2개 이상 소스), 장애 발생 시각, 영향 시스템 목록
예상 출력: 시스템별 분류된 로그 → 정렬된 이벤트 타임라인 → 근본 원인 → 재발 방지 액션 플랜
---

# 프롬프트 03: 타임라인 재구성 (프롬프트 체이닝 4단계)

## 사용 방법
각 단계(Step)를 **순서대로** 실행하고, 이전 단계의 출력을 다음 단계의 입력으로 사용합니다.

---

## STEP 1: 로그 분류 및 정규화

```
<Role>
당신은 시니어 SRE입니다. 여러 시스템에서 수집된 원시 로그를 분류하고 정규화하는 작업을 수행합니다.
</Role>

<Context>
- 장애 발생 시각: [YYYY-MM-DD HH:MM KST]
- 영향 시스템 목록: [예: API 서버, DB, 캐시(Redis), 로드밸런서]
- 분석 시간 범위: [장애 발생 30분 전 ~ 복구 후 10분]
</Context>

<RawLogs>
[여러 시스템에서 수집한 원시 로그를 여기에 붙여넣기]
</RawLogs>

<Task>
위 로그를 다음 기준으로 분류하고 정규화하세요:
1. 시스템별로 그룹화 (API 서버, DB, 캐시, 네트워크 등)
2. 각 로그 항목을 `[타임스탬프] [시스템] [레벨] [메시지]` 형식으로 통일
3. 분석 범위 외 로그는 제외
4. 중복 로그는 "반복 N회"로 압축
</Task>

<OutputFormat>
## 시스템별 분류 로그

### [시스템 A]
| 타임스탬프 | 레벨 | 메시지 |
|-----------|------|--------|
| | | |

### [시스템 B]
(동일 형식)

## 분류 후 주목할 이상 징후 요약
- [시스템 A]: [이상 신호 요약]
- [시스템 B]: [이상 신호 요약]
</OutputFormat>
```

---

## STEP 2: 이벤트 타임라인 재구성

> **이전 단계 출력을 `<ClassifiedLogs>`에 붙여넣어 실행**

```
<Role>
당신은 시니어 SRE입니다. 분류된 로그를 바탕으로 장애의 시간 순서를 재구성합니다.
</Role>

<ClassifiedLogs>
[Step 1 출력 결과를 여기에 붙여넣기]
</ClassifiedLogs>

<Task>
분류된 로그를 기반으로 시간 순서에 따른 이벤트 타임라인을 작성하세요.
- 정상 상태에서 장애 발생, 전파, 영향 확산, 복구까지의 흐름을 재구성
- 각 이벤트가 다른 시스템에 미친 연쇄 영향을 화살표(→)로 표현
- "최초 이상 징후 시각"을 명확히 표시
</Task>

<OutputFormat>
## 이벤트 타임라인

| 시각 | 시스템 | 이벤트 | 연쇄 영향 |
|------|--------|--------|---------|
| HH:MM | | | |
| HH:MM | ⚠️ **최초 이상** | | |
| HH:MM | | | → [다음 시스템]에 영향 |
| HH:MM | 🔴 서비스 영향 발생 | | |
| HH:MM | ✅ 복구 | | |

## 장애 전파 요약
[장애가 어느 시스템에서 시작하여 어떻게 전파되었는지 2~3문장으로 요약]
</OutputFormat>
```

## STEP 3: 근본 원인 분석

> **이전 단계 출력을 `<Timeline>`에 붙여넣어 실행**

```
<Role>
당신은 시니어 SRE입니다. 타임라인 분석을 바탕으로 근본 원인을 도출합니다.
반드시 첨부된 타임라인 데이터 내에서만 결론을 내리고, 불확실한 내용은 "데이터 부족"으로 표기하세요.
</Role>

<Timeline>
[Step 2 출력 결과를 여기에 붙여넣기]
</Timeline>

<Task>
타임라인을 기반으로:
1. 5-Whys 기법으로 근본 원인을 추적하세요 (프로세스/정책 수준까지)
2. 기여 요인을 식별하세요
3. 근본 원인과 기여 요인을 구분하여 명확히 표현하세요
</Task>

<OutputFormat>
## 근본 원인 분석

### 5-Whys
- Why 1: → 
- Why 2: → 
- Why 3: → 
- Why 4: → 
- Why 5: → 

### 근본 원인 (Root Cause)
> [1~2문장으로 명확하게]

### 기여 요인 (Contributing Factors)
1. 
2. 
3. 
</OutputFormat>
```

---

## STEP 4: 재발 방지 액션 플랜

> **이전 단계 출력을 `<RCAResult>`에 붙여넣어 실행**

```
<Role>
당신은 시니어 SRE입니다. RCA 결과를 바탕으로 실행 가능한 재발 방지 액션 플랜을 수립합니다.
</Role>

<RCAResult>
[Step 3 출력 결과를 여기에 붙여넣기]
</RCAResult>

<Task>
근본 원인과 기여 요인을 기반으로:
1. 즉시 조치 (24~48시간 내): 당장 위험을 줄이기 위한 임시 조치
2. 단기 개선 (1~2주): 재발을 방지하기 위한 기술적 개선
3. 장기 개선 (1~3개월): 시스템 근본적인 취약점 해소
각 항목에 담당 팀과 예상 기한을 포함하세요.
</Task>

<OutputFormat>
## 재발 방지 액션 플랜

### 즉시 조치 (24~48시간)
| 항목 | 내용 | 담당 | 기한 |
|------|------|------|------|

### 단기 개선 (1~2주)
| 항목 | 내용 | 담당 | 기한 |
|------|------|------|------|

### 장기 개선 (1~3개월)
| 항목 | 내용 | 담당 | 기한 |
|------|------|------|------|

### Post-mortem 보고 예정일
- [ ] 초안 작성: [날짜]
- [ ] 팀 리뷰: [날짜]
- [ ] 최종 공유: [날짜]
</OutputFormat>
```

## 활용 팁

- **각 단계 출력을 저장**한 뒤 다음 단계에 붙여넣는 것이 중요합니다.
- Step 3 완료 후 **06_quality_review.md**를 사용하여 최종 보고서를 검증하세요.
- 로그가 매우 방대한 경우 Step 1에서 핵심 이상 징후 로그만 우선 선별하여 입력하세요.
