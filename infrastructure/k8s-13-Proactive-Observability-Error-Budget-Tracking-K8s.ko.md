# Error Budget Tracking — 에러 버짓 추적

## 의미

에러 버짓 추적은 에러 버짓 추적을 수행할 수 없거나 에러 버짓 소진이 감지되었음을 나타냅니다(ErrorBudgetExhausted 또는 ErrorBudgetTrackingFailed 같은 알림 발생).
 에러 버짓 메트릭이 사용 불가하거나, 에러 버짓 추적 도구가 실패하거나, 에러 버짓 소진이 감지되거나, 에러 버짓 구성이 누락되거나, 에러 버짓 추적 모니터링에서 문제가 감지되는 것이 원인입니다. 에러 버짓 추적에서 실패가 나타나고, 에러 버짓 메트릭이 사용 불가하며, 에러 버짓 소진이 감지되고, 에러 버짓 추적이 실패합니다. 이는 성능 모니터링 계층과 에러 버짓 관리에 영향을 미치며, 일반적으로 에러 버짓 구성 실패, 에러 버짓 추적 도구 실패, 에러 버짓 메트릭 수집 문제, 또는 에러 버짓 추적 모니터링 격차로 인해 발생합니다. 에러 버짓 추적이 컨테이너 워크로드에 영향을 미치면 컨테이너 에러 버짓이 사용 불가하고 애플리케이션에서 에러 버짓 관리 실패가 발생할 수 있습니다.

## 영향

ErrorBudgetExhausted 알림 발생, ErrorBudgetTrackingFailed 알림 발생, 에러 버짓 추적 수행 불가, 에러 버짓 소진 감지, 에러 버짓 관리 저하 가능, 서비스 신뢰성 위험 가능. 에러 버짓 추적에서 실패가 나타나며, 에러 버짓 추적이 컨테이너 워크로드에 영향을 미치면 컨테이너 에러 버짓이 사용 불가하고, Pod 에러 버짓이 모니터링되지 않으며, 컨테이너 애플리케이션에서 에러 버짓 관리 실패가 발생할 수 있습니다. 애플리케이션에서 에러 버짓 추적 격차 또는 서비스 신뢰성 위험이 발생할 수 있습니다.

## 플레이북

1. 네임스페이스 <namespace>의 Deployment <deployment-name>을 전체 YAML 출력으로 조회하고 Service <service-name>을 상세 조회하여 서비스 구성과 SLO 정의를 확인합니다.

2. 네임스페이스 <namespace>의 최근 이벤트를 타임스탬프 순으로 조회하고 관련 오브젝트 이름 <deployment-name>으로 필터링하여 에러 버짓에 영향을 미치는 최근 문제를 확인합니다.

3. 서비스 `<service-name>`의 에러 버짓 구성을 조회하고 에러 버짓 목표 정의와 에러 버짓 계산 설정을 확인합니다.

4. 지난 30일 동안의 서비스 `<service-name>`에 대한 Prometheus 메트릭(error_rate, availability 포함)을 조회하여 에러 버짓 소비량과 잔여 버짓을 계산합니다.

5. 에러 버짓 추적 Pod 로그를 조회하고 지난 7일 이내의 에러 버짓 소진 패턴 또는 에러 버짓 추적 실패를 필터링합니다.

6. 에러 버짓 소진 감지 타임스탬프와 오류율 증가 타임스탬프를 1시간 이내로 비교하고, Prometheus 메트릭을 보조 증거로 사용하여 오류율 증가가 버짓 소진을 유발하는지 확인합니다.

7. 에러 버짓 추적 도구 실행 결과를 조회하고 도구 가용성 및 추적 완료 상태를 확인합니다.

8. 에러 버짓 구성 변경 타임스탬프와 에러 버짓 소비 변경 타임스탬프를 1시간 이내로 비교하고, 에러 버짓 구성 데이터를 보조 증거로 사용하여 구성 변경이 버짓 소비에 영향을 미치는지 확인합니다.

9. 서비스 `<service-name>`의 에러 버짓 소비 이력을 조회하고 에러 버짓 추세와 소비 정확도를 확인합니다.

10. 서비스 `<service-name>`의 Prometheus 알림을 조회하고 알림 에러 버짓 구성을 확인하여 에러 버짓 추적이 없는 알림을 식별합니다.

## 진단

1. 3단계의 에러 버짓 구성과 4단계의 Prometheus 메트릭을 검토합니다. 잔여 에러 버짓이 10% 미만이면 에러 버짓 소진이 즉각적인 주의가 필요한 주요 문제임을 확인합니다.

2. 4단계의 메트릭에서 오류율 또는 가용성 저하가 버짓 소비와 상관관계를 보이면 최근 서비스 신뢰성 문제가 버짓을 소진하고 있는 것입니다. 오류율이 안정적인데 버짓이 소비되면 SLO 목표가 기준 성능 대비 잘못 구성되었을 수 있습니다.

3. 7단계의 에러 버짓 추적 도구 결과를 분석합니다. 도구 실행에서 실패 또는 불완전 상태가 나타나면 추적 인프라 문제가 정확한 버짓 모니터링을 방해하고 있습니다. 도구가 정상이면 구성 분석으로 진행합니다.

4. 8단계의 구성 변경이 소비 변경과 상관관계를 보이면 최근 구성 수정이 버짓 계산에 영향을 미쳤을 수 있습니다. 최근 변경이 없으면 자연적인 서비스 저하가 원인일 가능성이 높습니다.

5. 10단계의 알림 커버리지에서 에러 버짓 추적이 없는 서비스가 있으면 모니터링 격차가 존재합니다. 알림이 구성되어 있지만 버짓 소진 시 발생하지 않으면 알림 임계값 조정이 필요합니다.

분석이 결론에 도달하지 못하는 경우: 9단계의 30일 소비 이력을 확인하여 버짓 소진이 예측 가능한 패턴을 따르는지(체계적 문제 시사) 또는 무작위로 발생하는지(인시던트 기반 소비 시사) 판단합니다. 에러 버짓 구성이 실제 서비스 동작과 일치하는지 확인하고 오류율 메트릭 수집이 정상 작동하는지 검증합니다.
