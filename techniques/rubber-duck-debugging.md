# Rubber Duck Debugging — 러버덕 디버깅

## 개념

문제를 고무 오리(또는 다른 대상)에게 한 줄씩 설명하다 보면
스스로 문제를 발견하게 되는 디버깅 기법.
'The Pragmatic Programmer'(1999)에서 대중화.
설명하는 행위 자체가 사고를 정리하고 암묵적 가정을 드러낸다.

## 핵심 원리

- 언어화 효과: 머릿속 생각을 말로 표현하면 논리적 빈틈이 드러남
- 느린 사고 강제: 코드를 "대충 훑는" 대신 한 줄씩 설명하면 System 2 사고 활성화
- 가정 노출: "당연하다"고 넘긴 부분을 설명하려 하면 가정이 드러남
- 비용 제로: 도구나 다른 사람 없이 혼자서 즉시 실행 가능

## 프롬프트 템플릿

```
다음 코드/시스템의 문제를 러버덕 디버깅으로 찾아주세요.

**문제 상황:** [버그/이슈 설명]
**기대 동작:** [어떻게 동작해야 하는지]
**실제 동작:** [실제로 어떻게 동작하는지]

**한 줄씩 설명하기:**

"이 코드의 목적은 [목적]이야."

"먼저 [1단계]를 하는데..."
- 입력: [예상 입력]
- 처리: [무엇을 하는지]
- 출력: [예상 출력]
→ 여기까지는 맞아? [검증]

"그 다음 [2단계]에서..."
- 입력: [이전 단계의 출력]
- 처리: [무엇을 하는지]
- 출력: [예상 출력]
→ 잠깐, 여기서 [의문점]이 생기네...

"아, [발견한 문제]가 원인이었어!"

**발견된 문제:** [구체적 문제]
**왜 놓쳤는가:** [어떤 가정이 잘못되었는지]
**수정 방안:** [해결책]
```

## 실전 예시

**시나리오: "사용자 목록 API가 빈 배열을 반환"**

"이 API는 DB에서 사용자를 조회해서 반환하는 거야."
"먼저 쿼리 파라미터에서 page와 limit을 파싱하는데... 기본값은 page=1, limit=20."
"그 다음 SQL 쿼리를 만드는데... OFFSET = (page-1) * limit = 0, LIMIT = 20."
"잠깐, 쿼리 결과를 변수에 담는데... rows = cursor.fetchall()..."
"아, fetchall() 전에 cursor.execute()를 호출하지 않았네!"
→ execute()와 fetchall() 사이에 commit()이 끼어들어 커서가 리셋됨

## 변형 및 조합

- **Rubber Duck + Feynman**: 초등학생에게 설명하듯 더 단순하게
- **Rubber Duck + CoT**: 설명 과정을 단계별 추론으로 구조화
- **Rubber Duck + Pair Programming**: 실제 동료에게 설명 (가장 효과적)

## 주의사항

- 문제가 코드 로직이 아닌 환경/설정에 있으면 효과 제한적
- 매우 큰 코드베이스에서는 어디부터 설명할지 결정이 어려움
- 이미 코드를 잘 아는 경우 "설명"이 형식적이 되어 효과 감소
- 동시성/비동기 버그는 순차적 설명으로 재현이 어려울 수 있음

## 적합한 상황

코드 디버깅, 로직 오류 찾기, 복잡한 코드 이해, 코드 리뷰, 온보딩
