# NodeDiskPressure-storage

## 의미

Node가 DiskPressure 조건에 진입했습니다(KubeNodeDiskPressure 알림 발생). 사용 가능한 디스크 공간 또는 inode가 Kubelet 축출 임계값 아래로 떨어져 Pod 축출이 발생하고 Node가 새로운 스케줄링에 안전하지 않게 되었습니다.
 디스크 압박으로 인해 Kubelet이 Pod를 축출하고 Node를 스토리지 리소스가 부족한 것으로 표시합니다.

## 영향

Node에서 Pod가 축출되고, 애플리케이션이 예기치 않은 재시작을 경험합니다. Node가 스케줄링 불가능해지며, 워크로드가 중단됩니다. 정상 종료 전에 Pod가 축출되면 데이터가 손실될 수 있습니다. KubeNodeDiskPressure 알림이 발생하고, Node 조건에 DiskPressure가 표시됩니다. Pod 축출 이벤트가 발생하며, 디스크 압박으로 인해 Node 스케줄링이 비활성화됩니다.

## 플레이북

1. Node <node-name>을 describe하여 DiskPressure 조건을 확인하고, 트리거된 임계값을 확인하며, 할당 가능한 리소스를 점검하고, Pod 축출을 보여주는 최근 이벤트를 검토합니다.

2. Node <node-name>의 이벤트를 타임스탬프 순으로 조회하여 디스크 압박이 시작된 시점, 어떤 Pod가 축출되었는지, 문제를 유발할 수 있는 Pod를 좁혀 파악합니다.

3. 모든 네임스페이스에서 Evicted 사유로 필터링된 이벤트를 타임스탬프 순으로 조회하여 축출된 Pod를 파악합니다. 이 Pod들은 가장 큰 디스크 소비자이거나 원인을 좁히는 데 도움이 되는 피해자일 가능성이 높습니다.

4. 영향받는 Node에 SSH로 접속하여 디스크 사용량을 확인합니다: 전체 파일시스템 사용량 및 inode 사용량(공간이 있어도 DiskPressure를 유발할 수 있음).

5. 영향받는 Node에서 /var/log를 확인하여 디스크 공간을 소비하는 축적된 로그를 확인합니다. 전체 로그 디렉토리 크기, Pod 로그, 가장 큰 로그 파일을 포함합니다.

6. 영향받는 Node에서 디스크를 소비하는 캐시된 컨테이너 이미지를 확인합니다: 모든 캐시된 이미지 목록, 전체 이미지 크기 계산, 런타임에 따른 컨테이너 스토리지 디렉토리 확인.

7. 영향받는 Node에서 디스크를 소비하는 고아 리소스를 확인합니다: 실행 중인 컨테이너와 비교한 미사용 이미지, 가비지 컬렉션되지 않은 종료된 컨테이너, 고아 Pod 디렉토리.

8. Kubelet 구성을 검사하여 가비지 컬렉션 설정을 확인합니다. imageGCHighThresholdPercent(기본값 85%) 및 imageGCLowThresholdPercent(기본값 80%)를 확인하여 축출 임계값을 이해합니다.

9. 영향받는 Node에서 로그 로테이션 문제를 확인하여 로테이션되지 않은 로그 및 저널 디스크 사용량을 파악합니다.

## 진단

1. 플레이북 2단계의 Node 이벤트를 분석하여 어떤 Pod가 축출되었는지 파악하고 주요 디스크 소비자를 좁힙니다. "DiskPressure" 사유의 "Evicted" 이벤트는 즉각적인 스토리지 고갈을 나타냅니다. 여러 Pod가 빠르게 연속으로 축출된 경우, 점진적 축적보다는 급성 디스크 고갈일 가능성이 높습니다.

2. 이벤트에서 특정 Pod가 대량 디스크 소비자로 식별되는 경우, 해당 Pod의 생성/재시작 타임스탬프를 플레이북 1단계의 DiskPressure 조건 전환 시간과 대조합니다. 디스크 압박이 시작되기 직전에 생성되거나 재시작된 Pod가 가장 큰 디스크 소비자일 가능성이 높습니다.

3. 이벤트에서 이미지 풀 활동(플레이북 2-3단계의 reason="Pulled")이 나타나는 경우, 이미지 풀 타임스탬프를 디스크 압박 시작과 대조하여 새 컨테이너 이미지가 문제를 유발했는지 판단합니다. DiskPressure 이전의 이미지 풀 급증은 컨테이너 이미지 캐시가 주요 소비자임을 시사합니다.

4. 이벤트에서 배포 또는 스케일링 활동이 나타나는 경우, 플레이북 5단계의 로그 디렉토리 크기와 교차 참조합니다. /var/log/pods에서 최근 배포된 Pod의 크기가 큰 경우, 애플리케이션 로그 축적이 원인일 가능성이 높습니다.

5. 이벤트가 결론을 내리기 어려운 경우, 플레이북 4-7단계의 디스크 사용량 분석을 확인합니다:
   - /var/lib/containers 또는 이미지 캐시(6단계)가 사용량을 지배하는 경우, 8단계의 Kubelet 가비지 컬렉션 임계값을 현재 사용량과 비교합니다.
   - /var/log가 지배하는 경우(5단계), 9단계의 로그 로테이션 상태를 확인합니다.
   - 고아 리소스가 존재하는 경우(7단계), 컨테이너 런타임 정리 실패가 원인입니다.

6. 단일 대량 소비자가 식별되지 않는 경우, 지난 1시간의 배포/스케일링 이벤트와 전체 디스크 사용량 추세를 비교하여 여러 소스에서의 점진적 축적을 파악합니다.

**지정된 시간 범위 내에서 상관관계를 찾을 수 없는 경우**: 검색 범위를 확장하고(5분 → 10분, 30분 → 1시간, 1시간 → 2시간), 더 긴 기간에 걸친 디스크 사용량 추세를 검토하여 점진적 증가를 파악합니다. 이미지 풀이나 로그 축적의 지연 효과를 확인하고, 컨테이너 런타임 스토리지 사용 패턴을 검사하며, 축출 임계값이 최근 수정되었는지 확인하고, 더 일찍 시작되었을 수 있는 백그라운드 프로세스나 로그 로테이션 실패를 확인합니다. 디스크 압박은 축출을 트리거하기 전에 점진적으로 축적될 수 있습니다.
