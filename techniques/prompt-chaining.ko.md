# Prompt Chaining — 프롬프트 연쇄

## 개념

복잡한 작업을 여러 단계의 프롬프트로 분해하고,
이전 단계의 출력을 다음 단계의 입력으로 연결하는 기법.
각 단계가 단순해져 정확도가 높아지고, 중간 결과를 검증할 수 있다.
LangChain, Semantic Kernel 등 프레임워크의 핵심 개념.

## 핵심 원리

- 복잡도 분산: 하나의 거대한 프롬프트보다 여러 작은 프롬프트가 정확
- 중간 검증: 각 단계의 출력을 확인하고 필요 시 수정 가능
- 재사용성: 각 체인을 독립적으로 테스트하고 재사용 가능
- 디버깅 용이: 어느 단계에서 문제가 발생했는지 특정 가능

## 프롬프트 템플릿

```
**전체 작업:** [복잡한 작업 설명]

**Chain 1 — [단계명] (예: 정보 수집):**
입력: [원본 데이터/질문]
지시: [이 단계에서 할 일]
출력 형식: [다음 단계가 사용할 수 있는 구조화된 형식]
→ 출력: [결과]

**Chain 2 — [단계명] (예: 분석):**
입력: Chain 1의 출력
지시: [이 단계에서 할 일]
출력 형식: [구조화된 형식]
→ 출력: [결과]

**Chain 3 — [단계명] (예: 결론 도출):**
입력: Chain 2의 출력
지시: [최종 처리]
→ 최종 출력: [결과]

**검증:** 최종 출력이 원래 작업 요구사항을 충족하는가?
```

## 실전 예시

**시나리오: 장애 보고서 자동 생성**

Chain 1 (로그 파싱): 원시 로그 → 구조화된 이벤트 타임라인
Chain 2 (원인 분석): 타임라인 → 5-Whys 분석 → 근본 원인
Chain 3 (영향 평가): 근본 원인 → 영향 범위, SLA 위반 여부
Chain 4 (보고서 생성): 분석 결과 → 경영진용 장애 보고서

각 단계가 독립적이므로 Chain 2만 교체하여 다른 분석 기법 적용 가능

## 변형 및 조합

- **Chaining + CoT**: 각 체인 내에서 CoT 적용
- **Chaining + Gate**: 중간에 조건 분기 (if-else 체인)
- **Chaining + Parallel**: 독립적인 체인을 병렬 실행 후 합류
- **Chaining + Human-in-the-Loop**: 중간 단계에서 사람 검증

## 주의사항

- 체인이 길어지면 에러 전파 위험 (앞 단계 오류가 뒤로 전파)
- 각 단계 간 인터페이스(출력 형식)를 명확히 정의해야 함
- 전체 레이턴시가 체인 수에 비례하여 증가
- 중간 단계의 출력이 예상과 다를 때 에러 핸들링 필요

## 적합한 상황

복잡한 분석, 문서 생성, 데이터 변환 파이프라인, 다단계 워크플로우, 자동화
