# Decomposition — 문제 분해 기법

## 개념

복잡한 문제를 관리 가능한 작은 하위 문제로 분해하는 기법.
분할 정복(Divide and Conquer)의 일반화.
각 하위 문제를 독립적으로 해결한 뒤 결합한다.
Polya(1945) "How to Solve It"에서 체계화된 문제 해결의 기본 원리.

## 핵심 원리

- 인지 부하 감소: 한 번에 하나의 작은 문제에만 집중
- 병렬화 가능: 독립적인 하위 문제는 동시에 해결 가능
- 재사용성: 하위 문제의 해결책을 다른 곳에서 재사용
- 테스트 용이: 작은 단위는 테스트하기 쉬움
- 진행 추적: 하위 문제 완료율로 전체 진행도 파악

## 프롬프트 템플릿

```
**복잡한 문제:** [전체 문제 설명]

**분해 전략:** [어떤 기준으로 분해하는가 — 기능별, 계층별, 시간순 등]

**분해 트리:**
```
[전체 문제]
├── [하위 문제 1] — [난이도: 상/중/하] [의존성: 없음/2에 의존]
│   ├── [세부 1-1]
│   └── [세부 1-2]
├── [하위 문제 2] — [난이도] [의존성]
│   ├── [세부 2-1]
│   └── [세부 2-2]
└── [하위 문제 3] — [난이도] [의존성]
    └── [세부 3-1]
```

**의존성 그래프:**
[어떤 하위 문제가 다른 것에 의존하는가]

**해결 순서 (의존성 + 우선순위 기반):**
1. [하위 문제 X] → 검증: [완료 기준]
2. [하위 문제 Y] → 검증: [완료 기준]
3. [하위 문제 Z] → 검증: [완료 기준]

**결합:** [하위 해결책들을 어떻게 합치는가]
**통합 검증:** [전체 문제가 해결되었는지 확인하는 방법]
```

## 실전 예시

**시나리오: "레거시 모놀리스에 API 게이트웨이 도입"**

분해:
├── 1. 현재 API 엔드포인트 인벤토리 작성 (의존성: 없음)
├── 2. 인증/인가 통합 방식 결정 (의존성: 1)
├── 3. 라우팅 규칙 설계 (의존성: 1)
├── 4. Rate Limiting 정책 수립 (의존성: 1)
├── 5. 게이트웨이 인프라 구축 (의존성: 2,3,4)
└── 6. 점진적 마이그레이션 (의존성: 5)

해결 순서: 1 → 2,3,4 (병렬) → 5 → 6

## 변형 및 조합

- **Decomposition + MECE**: 하위 문제를 MECE로 분류하여 누락/중복 방지
- **Decomposition + CoT**: 각 하위 문제를 CoT로 해결
- **Decomposition + Estimation**: 각 하위 문제의 소요 시간/비용 추정
- **Decomposition + Delegation**: 하위 문제를 팀원에게 분배

## 주의사항

- 분해 기준이 잘못되면 하위 문제 간 의존성이 복잡해짐
- 과도한 분해는 오버헤드 증가 (결합 비용)
- 하위 문제 간 인터페이스를 명확히 정의해야 함
- 전체를 보지 못하고 부분에 매몰될 수 있음 (숲 vs 나무)

## 적합한 상황

대규모 프로젝트 계획, 복잡한 버그 분석, 시스템 설계, 태스크 분배, 추정
