# 5 Whys — 다섯 번의 왜

## 개념

문제의 표면적 원인에 대해 '왜?'를 반복적으로 물어 근본 원인에 도달하는 기법.
도요타 생산 시스템(TPS)의 창시자 오노 다이이치(大野耐一)가 개발.
보통 5회 정도면 근본 원인에 도달하지만, 3회나 7회가 될 수도 있다.

## 핵심 원리

- 단순함: 특별한 도구 없이 "왜?"만 반복하면 됨
- 인과 관계 추적: 표면 증상에서 근본 원인까지 인과 체인을 따라감
- 조기 종료 방지: "왜?"를 계속 물어 피상적 원인에서 멈추지 않음
- 실행 가능한 원인 도달: 근본 원인은 "조치 가능한" 수준이어야 함

## 프롬프트 템플릿

```
**문제:** [관찰된 문제/증상]

**Why 1:** 왜 [문제]가 발생했는가?
→ 원인 1: [직접적 원인]

**Why 2:** 왜 [원인 1]이 발생했는가?
→ 원인 2: [한 단계 더 깊은 원인]

**Why 3:** 왜 [원인 2]가 발생했는가?
→ 원인 3: [더 깊은 원인]

**Why 4:** 왜 [원인 3]이 발생했는가?
→ 원인 4: [구조적/프로세스 원인]

**Why 5:** 왜 [원인 4]가 발생했는가?
→ 근본 원인: [조치 가능한 근본 원인]

**검증:** 근본 원인을 해결하면 최초 문제가 해결되는가? [예/아니오]
**역추적:** 근본 원인 → ... → 최초 문제 (인과 체인이 논리적인가?)

**대책:**
- 즉시 조치: [근본 원인 해결]
- 재발 방지: [프로세스/시스템 개선]
- 모니터링: [재발 감지 방법]
```

## 실전 예시

**시나리오: "프로덕션 배포 후 500 에러 급증"**

Why 1: 왜 500 에러가 발생? → 새 API 엔드포인트에서 NullPointerException
Why 2: 왜 NPE 발생? → 사용자 프로필 조회 시 null 반환을 처리하지 않음
Why 3: 왜 null 처리 누락? → 코드 리뷰에서 발견하지 못함
Why 4: 왜 코드 리뷰에서 놓침? → null safety 체크리스트가 없음
Why 5: 왜 체크리스트가 없음? → 코드 리뷰 프로세스에 표준 가이드라인 부재

근본 원인: 코드 리뷰 가이드라인 부재
대책: null safety 포함 코드 리뷰 체크리스트 작성 + lint 규칙 추가

## 변형 및 조합

- **5 Whys + Fishbone**: 각 Why에서 여러 원인이 나오면 Fishbone으로 분류
- **5 Whys + CoT**: 각 Why의 답을 CoT로 상세 추론
- **5 Whys + Hypothesis**: 각 단계에서 가설을 세우고 검증
- **5 Whys + MECE**: 각 단계의 가능한 원인을 MECE로 분류

## 주의사항

- 단일 인과 체인만 추적 — 복합 원인을 놓칠 수 있음 (Fishbone 보완)
- "왜?"의 답이 여러 개일 때 어떤 것을 따라갈지 판단 필요
- 5회가 항상 적절한 것은 아님 — 3회면 충분하거나 7회가 필요할 수 있음
- 비난(blame)이 아닌 시스템/프로세스 원인에 집중해야 함

## 적합한 상황

장애 분석(RCA), 버그 원인 추적, 프로세스 개선, 품질 관리, 포스트모템
