# Step-Back Prompting — 추상화 후 구체화

## 개념

구체적인 문제에 바로 답하지 않고, 먼저 한 단계 뒤로 물러나
상위 원리나 개념을 파악한 뒤 원래 문제에 적용하는 기법.
Zheng et al. (2023) 제안. "나무를 보기 전에 숲을 먼저 보라"는 접근.

## 핵심 원리

- 추상화의 힘: 구체적 디테일에 매몰되지 않고 상위 원리를 먼저 파악
- 전이 학습 효과: 상위 원리를 알면 유사한 구체적 문제에도 적용 가능
- MMLU, TimeQA 등 지식 집약적 벤치마크에서 유의미한 성능 향상
- 특히 물리, 화학 등 원리 기반 문제에서 효과적

## 프롬프트 템플릿

```
**원래 질문:** [구체적 질문]

**Step 1 — 한 발 물러서기 (Step-Back Question):**
"이 문제가 속한 상위 원리/개념/카테고리는 무엇인가?"
→ 상위 원리: [도출된 원리]

**Step 2 — 상위 원리 정리:**
- 핵심 원리 1: [...]
- 핵심 원리 2: [...]
- 관련 법칙/패턴: [...]

**Step 3 — 원리를 원래 문제에 적용:**
위 원리를 바탕으로 원래 질문에 답하면:
[구체적이고 원리에 기반한 답변]

**Step 4 — 검증:**
이 답변이 상위 원리와 일관되는가? [예/아니오, 근거]
```

## 실전 예시

**시나리오: "왜 Redis CLUSTER_DOWN 에러가 발생하는가?"**

Step-Back: "Redis Cluster의 가용성 보장 메커니즘은 무엇인가?"
→ 원리: 과반수 마스터가 살아있어야 클러스터 정상, 슬롯 할당이 완전해야 함

적용: 3개 마스터 중 2개가 다운 → 과반수 미달 → CLUSTER_DOWN
또는: 특정 슬롯 범위를 담당하는 노드가 다운 + 레플리카 없음

→ 원리 기반 진단이 증상 기반 진단보다 체계적

## 변형 및 조합

- **Step-Back + CoT**: 원리 파악 후 단계별 적용
- **Step-Back + First Principles**: 상위 원리를 제1원리까지 파고들기
- **Step-Back + Analogy**: 상위 원리를 다른 도메인에서 유추

## 주의사항

- 이미 원리를 잘 아는 문제에는 불필요한 우회
- Step-Back 질문 자체가 잘못되면 엉뚱한 방향으로 갈 수 있음
- 실무적/경험적 문제(원리가 명확하지 않은 경우)에는 효과 제한적

## 적합한 상황

도메인 지식이 필요한 문제, 원리 기반 추론, 일반화가 필요한 경우, 기술 면접, 교육
