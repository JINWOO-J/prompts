# 프롬프트 06: RCA 보고서 품질 검증 (CoVe + Self-Refinement)

## 사용 방법
`01_basic_rca.md` ~ `05_security_rca.md` 중 하나를 실행하여 초안을 얻은 뒤,  
해당 초안을 `<DraftRCA>`에 붙여넣고 이 프롬프트를 실행하세요.

---

## 프롬프트 본문 (Step 1: 검증 질문 생성)

```
<Role>
당신은 10년 경력의 시니어 SRE이자 엄격한 기술 문서 검토자입니다.
아래 RCA 초안을 기반으로, 이 보고서의 정확성과 완결성을 검증하기 위한
구체적인 검증 질문 목록을 생성하세요.
</Role>

<DraftRCA>
[이전 프롬프트 출력 결과(RCA 초안)를 여기에 붙여넣기]
</DraftRCA>

<Task>
이 RCA 초안의 다음 측면을 검증하기 위한 구체적인 질문 목록을 생성하세요:
1. 사실 정확성 (로그 데이터와 결론이 일치하는가?)
2. 논리적 일관성 (원인-결과 관계가 논리적으로 연결되는가?)
3. 완결성 (누락된 원인이나 기여 요인이 있는가?)
4. 실행 가능성 (재발 방지 권고안이 현실적으로 실행 가능한가?)
5. 명확성 (비전문가도 이해할 수 있는가?)
</Task>

<OutputFormat>
## 검증 질문 목록

### 사실 정확성
- Q1: [질문]
- Q2: [질문]

### 논리적 일관성
- Q3: [질문]
- Q4: [질문]

### 완결성
- Q5: [질문]
- Q6: [질문]

### 실행 가능성
- Q7: [질문]
- Q8: [질문]

### 명확성
- Q9: [질문]
- Q10: [질문]
</OutputFormat>
```

---

## 프롬프트 본문 (Step 2: 비판 및 개선점 도출)

> Step 1 검증 질문 목록을 `<VerificationQuestions>`에 붙여넣고 실행

```
<Role>
당신은 엄격한 시니어 SRE 검토자입니다.
아래 검증 질문에 답변하면서 RCA 초안의 약점과 개선이 필요한 부분을 식별하세요.
</Role>

<DraftRCA>
[초안을 다시 붙여넣기]
</DraftRCA>

<VerificationQuestions>
[Step 1의 검증 질문 목록을 붙여넣기]
</VerificationQuestions>

<Task>
각 검증 질문에 답변하고, 초안의 문제점을 식별하세요:
- 답변이 명확한 경우: ✅ [답변]
- 초안에서 확인 불가(데이터 부족): ⚠️ [무엇이 부족한가]
- 초안에서 오류 또는 논리 결함 발견: ❌ [어떤 오류이며 어떻게 수정해야 하는가]
</Task>

<OutputFormat>
## 검증 결과

| 질문 | 결과 | 비고 |
|------|------|------|
| Q1 | ✅ / ⚠️ / ❌ | |
| Q2 | | |
| ... | | |

## 발견된 주요 문제점 요약
1. ❌ **[문제점 제목]**: [설명 및 수정 방향]
2. ⚠️ **[데이터 부족 항목]**: [추가로 확인해야 할 사항]

## 개선 권고사항 (우선순위 순)
1. [가장 중요한 수정사항]
2. [두 번째 수정사항]
...
</OutputFormat>
```

## 프롬프트 본문 (Step 3: 최종 수정본 생성)

> Step 2 결과를 `<CritiqueResult>`에 붙여넣고 실행

```
<Role>
당신은 시니어 SRE입니다. 비판 검토 결과를 반영하여 RCA 보고서 최종본을 작성하세요.
수정 사항을 적용하되, 원본 RCA의 검증된 내용은 그대로 유지하세요.
</Role>

<DraftRCA>
[초안을 붙여넣기]
</DraftRCA>

<CritiqueResult>
[Step 2의 비판 결과를 붙여넣기]
</CritiqueResult>

<Task>
비판 검토 결과를 반영하여 최종 RCA 보고서를 작성하세요:
- 오류로 표시된 항목(❌) 수정
- 데이터 부족(⚠️) 항목은 "추가 확인 필요: [항목]"으로 명시
- 개선 권고사항을 보고서에 반영
- 전체 문서의 일관성과 명확성 향상
</Task>

<OutputFormat>
# RCA 최종 보고서 — [서비스명] 장애 / [날짜]

## 요약 (Executive Summary)
[비전문가도 이해할 수 있는 2~3문장 요약]

## 근본 원인
> [명확한 1~2문장]

## 장애 타임라인
[타임라인 표]

## 5-Whys 분석
[수정된 5-Whys]

## 재발 방지 액션 플랜
[수정된 액션 플랜 표]

## 추가 확인 필요 항목
- ⚠️ [항목 1]
- ⚠️ [항목 2]

## 보고서 메타데이터
- 작성일: [날짜]
- 작성자: [이름/팀]
- 검토자: [이름/팀]
- 버전: v1.0
</OutputFormat>
```

## 활용 팁

- **항상 마지막 단계로 실행**: 어떤 RCA 프롬프트를 사용했든 최종 보고서는 이 프롬프트를 통해 검증하세요.
- **Step 2가 가장 중요**: "비판 루프"는 AI 환각(Hallucination)과 논리적 비약을 잡아내는 핵심 단계입니다.
- **대외 공유 문서**: 경영진이나 고객에게 공유할 Post-mortem은 반드시 Step 3 최종본을 사용하세요.
