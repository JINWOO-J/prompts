# Role Prompting — 역할 부여 기법

## 개념

LLM에게 특정 전문가 역할을 부여하여 해당 관점에서 답변하도록 하는 기법.
역할에 맞는 전문 지식, 어조, 관점이 활성화되어 답변 품질이 향상된다.
System Prompt의 가장 기본적인 활용이며, 거의 모든 프롬프트의 시작점.

## 핵심 원리

- 지식 활성화: 특정 역할을 부여하면 관련 지식이 우선적으로 활성화
- 관점 고정: 일관된 관점에서 답변하여 맥락 전환 비용 감소
- 어조/스타일 제어: 전문가다운 어조, 초보자 친화적 설명 등 조절 가능
- 역할의 구체성이 핵심: "엔지니어"보다 "10년 경력 K8s 전문 SRE"가 효과적

## 프롬프트 템플릿

```
당신은 [역할/전문가 설명]입니다.

**전문 분야:** [구체적 전문 영역]
**경력:** [경력 수준과 주요 경험]
**관점:** [어떤 관점에서 답변해야 하는지]
**커뮤니케이션 스타일:** [기술적/비기술적, 상세/간결]

이 역할의 관점에서 다음 질문에 답하세요:

[질문]

답변 시 다음을 포함하세요:
- 전문가로서의 판단 근거
- 실무 경험에 기반한 조언
- 주의사항 및 리스크
- 추가로 확인해야 할 사항
```

## 실전 예시

**시나리오: 보안 감사**

"당신은 15년 경력의 클라우드 보안 아키텍트입니다. CISO에게 보고하는 입장에서
이 AWS 인프라의 보안 취약점을 평가하세요."

→ 역할이 구체적이므로: CISO 보고 형식, 리스크 매트릭스, 규정 준수 관점,
비즈니스 임팩트 중심의 답변이 나옴

vs "보안 전문가로서 평가하세요" → 일반적이고 기술 중심적인 답변

## 변형 및 조합

- **Role + CoT**: "시니어 SRE로서 단계별로 분석하세요"
- **Role + Six Hats**: 각 모자를 다른 역할로 수행
- **Role + Devil's Advocate**: "보안 감사관으로서 반론을 제기하세요"
- **Multi-Role**: 여러 역할이 토론하는 형식

## 주의사항

- 역할이 너무 구체적이면 답변 범위가 좁아질 수 있음
- LLM이 역할을 "연기"하면서 사실과 다른 답변을 할 수 있음 (과몰입)
- 역할 간 전환 시 이전 역할의 관점이 잔류할 수 있음
- 민감한 역할(의사, 변호사)은 면책 조항 필요

## 적합한 상황

전문 분야 상담, 코드 리뷰, 보안 감사, 다양한 관점 확보, 문서 작성 톤 조절
