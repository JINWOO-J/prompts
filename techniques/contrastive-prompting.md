# Contrastive Prompting — 대조 분석 기법

## 개념

두 가지 이상의 대상을 명시적으로 비교·대조하여
차이점과 공통점을 구조적으로 분석하는 기법.
선택지 간 트레이드오프를 명확히 드러내어 의사결정을 돕는다.
"좋은 예시 vs 나쁜 예시"를 보여주는 것도 대조 프롬프팅의 일종.

## 핵심 원리

- 차이 부각: 개별 분석보다 대조 분석이 차이점을 더 명확히 드러냄
- 트레이드오프 가시화: 각 선택지의 장단점이 나란히 보여 비교 용이
- 편향 감소: 하나만 분석하면 확증 편향, 대조하면 균형 잡힌 시각
- Good/Bad 예시: LLM에게 "이렇게 하라 / 이렇게 하지 마라"를 보여주면 정확도 향상

## 프롬프트 템플릿

```
다음 대상들을 체계적으로 대조 분석하세요.

**대상 A:** [첫 번째 옵션]
**대상 B:** [두 번째 옵션]
**(선택) 대상 C:** [세 번째 옵션]

**대조 분석:**

| 기준 | A | B | (C) | 비고 |
|------|---|---|-----|------|
| [기준 1 — 성능] | [...] | [...] | [...] | [...] |
| [기준 2 — 비용] | [...] | [...] | [...] | [...] |
| [기준 3 — 복잡도] | [...] | [...] | [...] | [...] |
| [기준 4 — 확장성] | [...] | [...] | [...] | [...] |
| [기준 5 — 생태계] | [...] | [...] | [...] | [...] |

**공통점:**
- [A와 B가 공유하는 특성]

**핵심 차이점:**
- [가장 중요한 차이 1]: [설명]
- [가장 중요한 차이 2]: [설명]

**상황별 추천:**
- [상황 X]에서는 A가 적합: [이유]
- [상황 Y]에서는 B가 적합: [이유]
- [상황 Z]에서는 C가 적합: [이유]

**결론:** [종합 판단]
```

## 실전 예시

**시나리오: "PostgreSQL vs MySQL vs MongoDB"**

| 기준 | PostgreSQL | MySQL | MongoDB |
|------|-----------|-------|---------|
| 데이터 모델 | 관계형+JSON | 관계형 | 문서형 |
| 복잡 쿼리 | 우수 (CTE, 윈도우) | 보통 | 제한적 |
| 쓰기 성능 | 보통 | 우수 | 우수 |
| 스키마 유연성 | 중간 (JSONB) | 낮음 | 높음 |

상황별: OLAP → PostgreSQL, 단순 CRUD → MySQL, 스키마 변경 잦음 → MongoDB

## 변형 및 조합

- **Contrastive + SWOT**: 각 대상을 SWOT으로 분석 후 대조
- **Contrastive + Hypothesis**: "A가 B보다 낫다"를 가설로 세우고 검증
- **Contrastive + Few-Shot**: 좋은 예시 vs 나쁜 예시를 대조로 제시

## 주의사항

- 비교 기준의 선택이 결과를 좌우 — 편향된 기준 주의
- 대상이 3개 이상이면 테이블이 복잡해짐 (2-3개가 적정)
- 정량적 비교가 어려운 기준(사용자 경험 등)은 주관적일 수 있음
- "승자"를 정하는 것이 목적이 아님 — 상황별 최적을 찾는 것

## 적합한 상황

기술 선택, 아키텍처 비교, 라이브러리 선정, 의사결정, 벤더 평가
