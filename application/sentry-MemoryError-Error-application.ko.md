# MemoryError-Error-application — 메모리 에러

## 의미

메모리 에러가 발생합니다 (Sentry 에러 이슈 트리거, 예외 유형 MemoryError). 애플리케이션이 메모리 부족 상태가 되어 메모리 할당이 사용 가능한 시스템 메모리를 초과할 때 애플리케이션 작업이 실패합니다. Sentry Issue Details 페이지에 "out of memory", "memory error", "OOM" 패턴의 에러 메시지가 표시되며, 스택 트레이스는 애플리케이션 코드 레이어를 가리키고, 에러 레벨은 Error 또는 Fatal 심각도를 나타냅니다. 이는 애플리케이션 레이어에 영향을 미치며, 일반적으로 메모리 누수, 과도한 메모리 사용, 또는 불충분한 메모리 할당으로 인한 메모리 관리 문제를 나타냅니다.

## 영향

Sentry 에러 이슈 알림이 발생하고, 메모리 에러가 발생하며, 애플리케이션 작업이 실패합니다. 애플리케이션이 크래시되거나 응답 불가 상태가 됩니다. 사용자에게 영향을 미치고, 이슈는 New 또는 Ongoing 상태로 유지됩니다. 애플리케이션 크래시가 발생하고, 영향받는 사용자 수가 증가하며, 에러 카운트가 지속적으로 증가합니다. Sentry 대시보드에서 High 우선순위로 표시되고, 애플리케이션 기능이 완전히 저하됩니다. OOM kill이 발생할 수 있습니다. 메모리 에러가 증가하고, 애플리케이션 성능이 저하됩니다.

## 플레이북

1. 이슈 유형(`metadata.type`), 우선순위(`priority`), 영향받는 사용자 수(`userCount`), 에러 메시지(`metadata.value`)를 포함한 이슈 상세 정보를 확인하여 "out of memory", "memory error", "OOM" 패턴의 MemoryError를 검증합니다.

2. 이슈 `<issue-id>`의 이벤트를 조회하고 이벤트 패턴을 분석합니다:
   - 최근 이벤트를 조회하여 스택 트레이스 프레임(`entries[].data.values[].stacktrace.frames[]`)을 검사하고 영향받는 애플리케이션 파일과 메모리 집약적 작업을 식별합니다
   - 최근 24시간 동안의 이벤트 빈도(일정, 급증, 점진적 증가)를 분석하고 이벤트 데이터에서 사용자 영향 분포(전체 사용자 vs 특정 세그먼트)를 확인합니다

3. 에러 메시지(`metadata.value`)에서 메모리 할당 크기와 에러 컨텍스트를 포함한 메모리 에러 상세 정보를 추출하여 메모리 집약적 작업을 식별합니다.

4. 이슈 상세 정보의 릴리스 타임라인(firstRelease vs lastRelease)을 비교하여 이슈가 배포와 연관되는지 확인합니다.

5. `firstRelease.lastCommit`이 존재하면 `firstRelease.versionInfo.package`에서 리포지토리 이름을 추출하거나 서비스 매핑을 사용한 후, GitHub 커밋 상세 정보를 조회하고 커밋이 PR에 포함되어 있는지 확인합니다. PR이 발견되면 PR diff를 조회하여 메모리 누수 또는 과도한 메모리 사용을 도입할 수 있는 코드 변경을 분석합니다.

6. Sentry의 에러 컨텍스트가 불충분하면, 에러 타임스탬프 전후의 서비스 ELK 로그를 검색하여 메모리 관련 로그, OOM kill, 메모리 할당 패턴을 식별합니다.

7. 프로젝트 `<project-name>`의 유사 이슈를 조회하고 공통 릴리스 또는 배포 타임스탬프를 공유하는지 확인하여 광범위한 인시던트를 식별합니다.

## 진단

1. 플레이북 2단계의 에러 이벤트 패턴을 분석하여 에러 추세를 식별합니다. 이벤트가 배포와 연관된 급격한 급증을 보이면 메모리 문제를 도입한 배포 관련 코드 변경을 나타냅니다. 에러가 일정하면 지속적인 메모리 누수 또는 부족한 메모리 할당을 나타냅니다. 에러가 간헐적이면 부하 의존적 메모리 고갈 또는 과도한 메모리 사용을 유발하는 특정 입력 패턴을 나타냅니다.

2. 에러 급증이 배포와 연관되면(플레이북 4단계 릴리스 타임라인), `firstRelease` 타임스탬프와 에러 발생 시점을 비교하여 배포가 트리거인지 확인합니다. `firstRelease`와 `lastRelease`가 다르면 메모리 에러가 초기 릴리스에서 시작되었는지 후속 릴리스에서 증가했는지 분석합니다.

3. `firstRelease.lastCommit`이 존재하고 이벤트가 코드 관련 이슈를 나타내면(플레이북 5단계), GitHub 커밋 변경 사항을 분석하여 대규모 객체 할당, 무제한 컬렉션, 또는 메모리 누수 패턴을 도입하는 특정 코드 변경을 식별합니다.

4. 이벤트가 인프라 관련 문제(점진적 메모리 증가, OOM kill — 플레이북 3단계)를 나타내면, Sentry 이벤트의 에러 타임스탬프 전후의 ELK 로그(플레이북 6단계)에서 연관된 메모리 사용 추세, 컨테이너 OOM 이벤트, 또는 애플리케이션 재시작 패턴을 검색합니다.

5. 명확한 배포 연관성이 없으면, 플레이북 2단계 이벤트 데이터에서 영향받는 사용자 패턴을 분석합니다. 모든 사용자가 영향받으면 메모리 누수 또는 불충분한 메모리 할당일 가능성이 높습니다. 특정 세그먼트가 영향받으면 메모리 고갈을 유발하는 특정 입력 패턴 또는 데이터 크기일 가능성이 높습니다.

6. 유사 이슈가 존재하고(플레이북 7단계) 공유된 릴리스 타임스탬프가 있으면, 여러 서비스에 걸쳐 메모리 문제를 도입한 코드 변경이 있는 광범위한 배포 인시던트를 나타냅니다.
