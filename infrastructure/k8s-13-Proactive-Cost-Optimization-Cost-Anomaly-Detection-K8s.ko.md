# Cost Anomaly Detection (비용 이상 감지)

## 의미

비용 이상 감지 알림은 Kubernetes 리소스 비용이 정상 지출 베이스라인에서 벗어나는 예기치 않은 급증이나 비정상 패턴을 보이는 상황(CostAnomalyDetected 또는 UnexpectedCostIncrease 같은 알림 발생)을 나타냅니다. 비용 메트릭이 이력 베이스라인 초과, 비용 추세의 급격한 증가, 비용 할당의 예기치 않은 리소스 사용, 예산 임계값 초과, 또는 비용 패턴의 비정상 리소스 소비 등이 원인입니다.
 비용 메트릭이 급증이나 이상을 보이고, 예산 알림이 예기치 않게 발생하며, 비용 추세가 이력 패턴에서 벗어나고, 비용 할당이 예기치 않은 리소스 사용을 나타냅니다. 이는 비용 관리 계층과 재무 모니터링에 영향을 미치며, 주로 리소스 프로비저닝 급증, 리소스 사용 증가, 가격 변경, 또는 비용 할당 문제가 원인입니다. 비용 이상이 컨테이너 워크로드에 영향을 미치는 경우, 컨테이너 리소스 소비가 예기치 않게 증가하고 애플리케이션이 비용 초과를 겪을 수 있습니다.

## 영향

CostAnomalyDetected 알림 발생, UnexpectedCostIncrease 알림 발생, 예산 임계값 초과, 예기치 않은 비용 누적, 비용 추세가 이력 패턴에서 이탈, 재무 계획 차질. 비용 메트릭이 급격한 급증이나 이상을 보입니다. 비용 이상이 컨테이너 워크로드에 영향을 미치는 경우, 컨테이너 리소스 소비가 예기치 않게 증가하고, Pod 리소스 사용이 급증하며, 컨테이너 애플리케이션이 비용 초과를 겪을 수 있습니다. 애플리케이션이 예기치 않은 비용 증가나 예산 초과를 경험할 수 있습니다.

## 플레이북

1. namespace <namespace>에서 Pod를 wide 출력으로 조회하고 Pod 리소스 사용량을 확인하여 워크로드 전체의 현재 리소스 소비를 식별합니다.

2. namespace <namespace>에서 최근 이벤트를 타임스탬프 순으로 조회하여 비용 이상을 유발할 수 있는 최근 스케일링 이벤트나 리소스 변경을 확인합니다.

3. namespace <namespace>에서 Pod를 describe하여 비용 분석을 위한 리소스 요청과 제한을 파악합니다.

4. namespace <namespace>에서 리소스 쿼터를 describe하여 namespace 수준의 리소스 소비를 분석합니다.

5. 리소스 사용량의 Prometheus 메트릭(CPU, 메모리 소비)을 최근 7일 동안 조회하여 예기치 않은 사용 증가가 있는 리소스를 식별합니다.

6. namespace <namespace>의 리소스 유형별 비용 할당 데이터를 최근 7일 동안 조회하고 이력 평균 대비 예기치 않은 비용 증가가 있는 리소스를 식별합니다.

7. 비용 모니터링 서비스에서 비용 이상 감지 결과를 조회하고 영향받는 리소스와 비용 영향을 포함한 이상 세부 정보를 확인합니다.

8. 비용 급증 타임스탬프와 리소스 프로비저닝 타임스탬프를 비교하여 비용 증가가 리소스 생성 이벤트와 상관관계가 있는지 확인하고, 비용 메트릭을 보조 증거로 활용합니다.

## 진단

1. 1단계와 3단계의 Pod 리소스 사용량을 검토합니다. 특정 Pod가 이력 평균보다 현저히 높은 리소스 소비를 보이면 이것이 비용 이상의 원인일 가능성이 높습니다. 워크로드를 식별하고 사용 증가를 조사합니다.

2. 5단계의 7일 리소스 사용 메트릭을 분석합니다. 사용 급증이 비용 급증과 상관관계가 있으면 실제 리소스 소비가 비용을 유발하고 있습니다. 사용이 안정적인데 비용이 증가했으면 가격 변경이나 새로운 리소스 유형이 원인일 수 있습니다.

3. 6단계의 비용 할당 데이터에서 특정 리소스나 namespace에 예기치 않은 비용 증가가 있으면 해당 영역에 조사를 집중합니다. 비용 증가가 분산되어 있으면 클러스터 전체 요인이 원인입니다.

4. 7단계의 비용 이상 결과를 검토합니다. 이상 감지가 특정 리소스나 시간대를 식별했으면 해당 세부 정보를 활용하여 조사를 안내합니다.

5. 8단계의 리소스 프로비저닝 분석에서 비용 급증 시점에 새 리소스가 생성되었으면 프로비저닝 이벤트가 비용을 유발하고 있습니다. 이러한 프로비저닝 이벤트가 의도적이었는지 확인합니다.

분석이 결론에 이르지 못하면: 2단계의 이벤트에서 스케일링 이벤트나 리소스 변경을 확인합니다. 4단계의 리소스 쿼터를 검토하여 namespace 수준의 소비를 파악합니다. 비용 이상이 일회성 이벤트인지(프로비저닝 또는 인시던트 시사) 또는 지속적인지(최적화가 필요한 사용 패턴 변화 시사) 판단합니다.
