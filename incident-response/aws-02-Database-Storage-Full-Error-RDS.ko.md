# RDS Storage Full Error - RDS 스토리지 가득 참 오류

## 의미

RDS 데이터베이스 스토리지가 가득 차거나 용량 제한에 근접하는 현상(FreeStorageSpace와 같은 경보 트리거)은 할당된 스토리지가 부족하거나, 대형 테이블이 과도한 공간을 소비하거나, 오래되거나 불필요한 데이터가 축적되거나, 자동 스토리지 스케일링이 비활성화되었거나, 스토리지 지표가 용량 소진을 나타내거나, RDS 로그 파일이 과도한 공간을 소비할 때 발생합니다.
 데이터베이스 쓰기 작업이 실패하고, 스토리지 가득 참 오류가 발생하며, CloudWatch 지표에서 FreeStorageSpace가 0에 근접합니다. 이는 데이터베이스 계층에 영향을 미치며 데이터 쓰기를 차단합니다. 일반적으로 스토리지 할당 문제, 데이터 증가 또는 로그 파일 축적이 원인이며, RDS Aurora를 사용하는 경우 공유 스토리지 모델이 다르게 동작할 수 있고 스토리지가 차면서 스토리지 I/O 성능이 저하될 수 있습니다.

## 영향

데이터베이스 쓰기 작업이 실패합니다. 스토리지 가득 참 오류가 발생합니다. FreeStorageSpace 경보가 발생합니다. 데이터베이스가 읽기 전용이 됩니다. 새 데이터를 삽입할 수 없습니다. 데이터베이스 백업이 실패할 수 있습니다. 스토리지 용량이 소진됩니다. 데이터베이스 성능이 저하됩니다. 자동 스케일링이 트리거될 수 있습니다. 애플리케이션 쓰기 작업에 오류가 발생합니다. 데이터베이스 쿼리가 타임아웃됩니다. 트랜잭션 실패가 발생합니다. RDS Aurora를 사용하는 경우 공유 스토리지가 차면서 스토리지 I/O 성능이 저하됩니다. 스토리지 소진으로 인해 애플리케이션에서 오류나 성능 저하가 발생할 수 있으며, 커넥션 풀 제한에 도달하여 새 연결이 방해될 수 있습니다.

## 플레이북

1. RDS 인스턴스 `<rds-instance-id>`가 존재하고 "available" 상태인지, 리전 `<region>`의 RDS AWS 서비스 상태가 정상인지 확인합니다.
2. RDS 인스턴스 `<rds-instance-id>`의 CloudWatch 지표(FreeStorageSpace)를 조회하여 현재 스토리지 사용률을 확인하고, 최근 7일간의 스토리지 추세를 분석합니다.
3. RDS 인스턴스 `<rds-instance-id>`의 스토리지 구성을 조회하여 할당된 스토리지 설정과 자동 스토리지 스케일링 구성을 확인하고, 최대 스토리지 제한을 점검합니다.
4. RDS 인스턴스 로그가 포함된 CloudWatch Logs 로그 그룹에서 용량 문제나 스토리지 오류를 나타내는 스토리지 관련 메시지를 필터링하여 로그 파일 크기 정보를 확인합니다.
5. RDS 인스턴스 `<rds-instance-id>`의 로그 파일 보존 설정을 조회하여 로그 파일 보존 기간을 확인하고, 로그 파일이 과도한 스토리지를 소비하고 있는지 점검합니다.
6. RDS 인스턴스 `<rds-instance-id>`의 자동 스토리지 스케일링 설정을 조회하여 향후 문제를 방지하기 위해 자동 스케일링이 활성화되어 있는지 확인하고, 스케일링 트리거 임계값을 점검합니다.
7. RDS 인스턴스 `<rds-instance-id>`의 백업 보존 및 Performance Insights 데이터 보존 설정을 조회하여 자동 백업 보존 기간과 Performance Insights 데이터 보존 기간을 확인하고, 백업이나 Performance Insights 데이터가 스토리지를 소비하고 있는지 점검합니다.
8. RDS Aurora를 사용하는지 확인하고 Aurora 스토리지 모델(공유 스토리지)을 점검하며, RDS 인스턴스 `<rds-instance-id>`의 CloudWatch 지표(ReadIOPS, WriteIOPS)를 조회하여 표준 RDS와의 스토리지 동작 차이를 확인하고 스토리지가 차면서 스토리지 I/O 성능 저하가 있는지 점검합니다.

## 진단

1. 플레이북 1단계의 AWS 서비스 상태를 분석하여 리전의 RDS 서비스 가용성을 확인합니다. 서비스 상태에 문제가 있으면 스토리지 오류는 구성 변경이 아닌 모니터링이 필요한 AWS 측 문제일 수 있습니다.

2. 플레이북 2단계의 FreeStorageSpace 지표에서 스토리지가 0에 근접하거나 0이면 스토리지 소진이 확인됩니다. 7일간 추세를 분석하여 증가가 점진적(데이터 축적)인지 갑작스러운(대량 작업, 로그 폭발)인지 판단합니다.

3. 플레이북 3단계의 스토리지 구성에서 할당된 스토리지가 부족하고 자동 스토리지 스케일링이 비활성화되어 있으면 스토리지가 자동으로 확장될 수 없습니다. 향후 소진을 방지하기 위해 적절한 최대 제한으로 자동 스케일링을 활성화합니다.

4. 플레이북 4단계의 RDS 로그에서 스토리지 관련 오류 메시지가 표시되면 스토리지 오류가 처음 나타난 시점을 파악합니다. 이 타임스탬프가 스토리지가 심각하게 부족해진 시점을 표시합니다.

5. 플레이북 5단계의 로그 파일 보존에서 보존 기간이 길면 로그 파일이 상당한 스토리지를 소비할 수 있습니다. 로그 파일이 전체 스토리지 사용량의 큰 부분을 차지하는지 확인합니다.

6. 플레이북 6단계의 자동 스케일링 설정에서 자동 스케일링이 활성화되어 있지만 최대 스토리지 임계값에 도달한 경우, 구성된 최대값이 부족합니다. 최대 스토리지 할당을 늘립니다.

7. 플레이북 7단계의 백업 보존이 많은 일수로 설정되어 있으면 자동 백업이 추가 스토리지를 소비합니다. Performance Insights 보존이 연장되어 있으면 해당 데이터도 스토리지를 소비합니다.

8. RDS Aurora를 사용하는 경우(플레이북 8단계), 스토리지는 클러스터 전체에서 공유되며 자동으로 확장됩니다. 클러스터가 커지면서 스토리지 I/O 성능이 저하될 수 있습니다. 스토리지 압박을 나타내는 I/O 병목 현상에 대해 ReadIOPS 및 WriteIOPS 지표를 확인합니다.

상관관계를 찾을 수 없는 경우: FreeStorageSpace 지표 분석을 30일로 확장하고, 대형 테이블을 식별하기 위해 데이터베이스 테이블 크기를 점검하고, 공간을 소비하는 고아 또는 임시 테이블을 확인하고, 바이너리 로그 보존 설정을 검토합니다. 스토리지 소진은 임시 테이블을 생성하는 최적화되지 않은 쿼리, 대형 트랜잭션 로그 또는 테이블 재구축이 필요한 스키마 변경으로 인해 발생할 수 있습니다.
