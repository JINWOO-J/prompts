# Fishbone (Ishikawa) — 원인-결과 다이어그램

## 개념

문제(결과)를 머리로, 원인 카테고리를 뼈대로 하는 다이어그램.
이시카와 카오루(1968) 제안. 6M(Man, Machine, Method, Material, Measurement, Mother Nature)
카테고리로 원인을 체계적으로 분류. 복합 원인 분석에 5 Whys보다 효과적.

## 핵심 원리

- 다차원 분석: 단일 인과 체인이 아닌 여러 카테고리에서 동시에 원인 탐색
- 시각적 구조: 다이어그램 형태로 원인 관계를 한눈에 파악
- 팀 브레인스토밍에 적합: 여러 사람이 각 카테고리에 원인을 추가
- MECE와 자연스럽게 결합: 6M 카테고리가 MECE 구조

## 프롬프트 템플릿

```
**문제 (Fish Head):** [문제 설명]

**원인 분석 (Fish Bones):**

**1. People (사람/팀):**
- [원인 1-1]: [상세 설명 — 기술 수준, 커뮤니케이션, 인력 부족 등]
- [원인 1-2]: [상세]

**2. Process (프로세스/방법):**
- [원인 2-1]: [상세 — 절차 부재, 비효율적 워크플로우 등]
- [원인 2-2]: [상세]

**3. Technology (기술/도구):**
- [원인 3-1]: [상세 — 소프트웨어 버그, 하드웨어 한계, 도구 부족 등]
- [원인 3-2]: [상세]

**4. Environment (환경):**
- [원인 4-1]: [상세 — 인프라, 네트워크, 외부 서비스 등]

**5. Measurement (측정/모니터링):**
- [원인 5-1]: [상세 — 메트릭 부재, 알림 미설정, 가시성 부족 등]

**6. Data/Material (데이터/자원):**
- [원인 6-1]: [상세 — 데이터 품질, 리소스 부족 등]

**원인 우선순위:**
| 순위 | 원인 | 카테고리 | 영향도 | 조치 가능성 |
|------|------|---------|--------|-----------|
| 1 | [...] | [...] | 높/중/낮 | 높/중/낮 |
| 2 | [...] | [...] | | |
| 3 | [...] | [...] | | |

**가장 유력한 근본 원인:** [선택 및 근거]
```

## 실전 예시

**시나리오: "배포 실패율 30%"**

People: 배포 담당자 교육 부족, 온콜 인수인계 미흡
Process: 배포 체크리스트 없음, 롤백 절차 미정의
Technology: CI/CD 파이프라인 불안정, 테스트 커버리지 낮음
Environment: 스테이징과 프로덕션 환경 차이
Measurement: 배포 성공률 메트릭 미수집
Data: 설정 파일 환경별 분리 안 됨

→ 우선순위 1: CI/CD 파이프라인 안정화 + 테스트 강화

## 변형 및 조합

- **Fishbone + 5 Whys**: 각 뼈대의 원인에 5 Whys 적용하여 근본 원인 도달
- **Fishbone + MECE**: 카테고리를 MECE로 검증
- **Fishbone + Prioritization Matrix**: 원인별 영향도×조치가능성 매트릭스

## 주의사항

- 카테고리가 고정적이라 도메인에 맞지 않을 수 있음 (커스텀 카테고리 사용)
- 원인 간 상호작용(교차 영향)을 표현하기 어려움
- 브레인스토밍 없이 혼자 하면 편향된 원인만 나열할 수 있음
- 원인 나열에 그치고 검증/우선순위 없이 끝나면 무의미

## 적합한 상황

복합적 원인 분석, 장애 RCA, 품질 문제, 프로세스 개선, 팀 회고
